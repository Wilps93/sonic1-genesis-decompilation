### 1.1 Чит-код Level Select

| | |
|---|---|
| **Адрес ROM** | `$002EE0–$002F16` |
| **Файл** | sonic1_pseudo_decompilation.c, строки 1468–1494 |
| **Функция** | `EntryPoint` (обработчик титульного экрана) |

После появления логотипа SEGA (звук `$E4`) запускается **30-кадровое окно ввода** (~0.5 секунды):

```c
/* 002EE0 */      d0 = #$e1;                    // музыка Title Screen
/* 002EE4 */      PlaySound_Special();
/* 002EF2 */      $f614.w = #$1e;               // таймер = 30 кадров (0.5 сек)
  loc_002EF8:
/* 002EF8 */      $f62a.w = #$2;
/* 002EFE */      WaitForVBlank();
/* 002F02 */      test($f614.w);                 // таймер ещё тикает?
/* 002F06 */      if (Z == 1) goto loc_002F10;   // истёк → перейти к Title Screen
/* 002F08 */      $f605.w &= #$80;              // проверка кнопки Start (бит 7)
/* 002F0E */      if (Z == 1) goto loc_002EF8;   // Start не нажат → ждать
  loc_002F10:
/* 002F10 */      $f600.w = #$4;                 // game mode = Title Screen
```

**Как активировать:**
1. На экране SEGA ввести: **Up, Down, Left, Right**
2. Дождаться титульного экрана
3. Зажать **A** и нажать **Start**
4. Откроется Level Select вместо старта игры

> Обработчик последовательности D-pad находится внутри массива данных `GameModeArray` (27528 байт), который декомпилятор интерпретировал как код — фактически это jump-table + data.

### 1.2 Активация Debug Mode

| | |
|---|---|
| **Адрес ROM** | `$003460–$003570` |
| **Файл** | sonic1_pseudo_decompilation.c, строки 7697–7790 |
| **Функция** | `LevelSelect` |

В меню Level Select последняя строка (позиция `$14` = 20) — **Sound Test**:

```c
/* 0034AA */      compare($ff82.w, #$14);       // курсор на Sound Test?
/* 0034B0 */      if (Z == 0) goto loc_0034E4;  // нет → обычная навигация
/* 0034BC */      d0 = $ff84.w;                 // значение звука
/* 00355A */      d0 = $ff84.w;
/* 00355E */      d0 += #$80;                   // ID = значение + $80
```

**Переменные:**
- `$FF80` — таймер авто-повтора (сбрасывается в `$0B` = 11 кадров)
- `$FF82` — позиция курсора: `$00`–`$14` (21 строка, `$14` = Sound Test)
- `$FF84` — значение Sound Test: `$00`–`$4F` (отображается как `$80`–`$CF`)

**Как активировать:**
1. Войти в Level Select (см. чит выше)
2. Опуститься до строки **Sound Test**
3. Выбрать звук и активировать нужную комбинацию (C + Start)
4. Флаг `$FE08` становится ≠ 0

### 1.3 Slow Motion / Покадровый просмотр

| | |
|---|---|
| **Адрес ROM** | `$0013A8–$001412` |
| **Файл** | sonic1_pseudo_decompilation.c, строки 6034–6080 |
| **Функция** | `PauseGame` |

Скрытая функция покадрового воспроизведения внутри паузы:

```c
/* 0013A8 */      test($fe12.w);                 // пауза активна?
/* 0013AE */      if (Z == 1) goto loc_00140A;   // нет → пропуск
/* 0013B6 */      test_bit($f605.w, #$7);        // Start нажат?
/* 0013BC */      if (Z == 1) goto loc_001410;   // → снять паузу
  loc_0013CA:
/* 0013CA */      $f62a.w = #$10;                // ожидание VBlank
/* 0013D0 */      WaitForVBlank();
/* 0013D4 */      test($ffe1.w);                 // демо-режим?
/* 0013D8 */      if (Z == 1) goto loc_0013FC;   // да → другая логика
/* 0013DA */      test_bit($f605.w, #$6);        // A нажата (в демо)?
/* 0013E0 */      if (Z == 1) goto loc_0013EC;
/* 0013E2 */      $f600.w = #$4;                 // → к титульному экрану
  loc_0013EC:
/* 0013EC */      test_bit($f604.w, #$4);        // ═══ B ЗАЖАТА? ═══
/* 0013F2 */      if (Z == 0) goto loc_001412;   // нет → проверка Start
/* 0013F4 */      test_bit($f605.w, #$5);        // ═══ C НАЖАТА? ═══
/* 0013FA */      if (Z == 0) goto loc_001412;   // ДА → ОДИН КАДР ВПЕРЁД!
  loc_0013FC:
/* 0013FC */      test_bit($f605.w, #$7);        // Start нажат?
/* 001402 */      if (Z == 1) goto loc_0013CA;   // нет → цикл ожидания
```

**Управление на паузе:**

| Кнопка | Действие |
|---|---|
| **Start** | Снять паузу |
| **A** (в демо) | Вернуться на титульный экран |
| **Зажать B + Нажать C** | ⚡ **Продвинуть игру на 1 кадр** |

> Эта функция — инструмент разработчика, оставленный в релизной версии. Позволяет покадрово анализировать физику и столкновения.

### 1.4 Debug Placement Mode (режим размещения объектов)

| | |
|---|---|
| **Адрес ROM** | `$01CFD6–$01D16C` |
| **Файл** | sonic1_pseudo_decompilation.c, строки 32720–32870 |
| **Функция** | `DebugMode` / `DebugMode_Sub` |

Полноценный редактор объектов прямо во время игры:

```c
// ═══ ПЕРЕМЕЩЕНИЕ ═══
/* 01CFE6 */      d4 = $f604.w;                  // зажатые кнопки
/* 01D000 */      $fe0a.w -= #$1;                // таймер ускорения
/* 01D006 */      $fe0a.w = #$1;                 // сброс
/* 01D00C */      $fe0b.w += #$1;                // увеличить скорость
/* 01D012 */      $fe0b.w = #$ff;               // макс = 255

// ═══ НАПРАВЛЕНИЯ ═══
/* 01D030 */      test_bit(d4, #$0);             // Up?    → d2 -= скорость
/* 01D03C */      test_bit(d4, #$1);             // Down?  → d2 += скорость
/* 01D052 */      test_bit(d4, #$2);             // Left?  → d3 -= скорость
/* 01D05E */      test_bit(d4, #$3);             // Right? → d3 += скорость

// ═══ ПЕРЕКЛЮЧЕНИЕ ОБЪЕКТОВ ═══
/* 01D076 */      test_bit($f605.w, #$5);        // C + A → следующий
/* 01D08A */      test_bit($f605.w, #$6);        // B + A → предыдущий
/* 01D092 */      $fe06.w += #$1;                // индекс объекта ++
/* 01D09C */      $fe06.w = #$0;                 // wrap to 0

// ═══ РАЗМЕЩЕНИЕ ОБЪЕКТА ═══
/* 01D0A6 */      test_bit($f605.w, #$5);        // C (без A)?
/* 01D0AE */      FindFreeObj();                  // найти свободный слот
/* 01D0B4 */      if (Z == 0) goto loc_01D0EA;   // нет слотов → пропуск
/* 01D0B6 */      $8(a1) = $8(a0);               // X позиция
/* 01D0BC */      $c(a1) = $c(a0);               // Y позиция
/* 01D0C2 */      $0(a1) = $4(a0);               // ID объекта

// ═══ ВЫХОД ИЗ DEBUG MODE ═══
/* 01D0EA */      test_bit($f605.w, #$4);        // B (без A)?
/* 01D0F4 */      $fe08.w = d0;                  // d0 = 0 → ВЫКЛЮЧИТЬ debug
```

**Полное управление:**

| Вход | Действие |
|---|---|
| **D-pad** | Свободное перемещение по уровню |
| **A + C** | Переключить на следующий объект |
| **A + B** | Переключить на предыдущий объект |
| **C** (без A) | Поставить текущий объект |
| **B** (без A) | Выйти из Debug Mode |

**Ускорение:** переменные `$FE0A` (таймер) и `$FE0B` (скорость, макс `$FF`) — чем дольше зажато направление, тем быстрее движение.

### 1.5 Эффекты Debug Mode (22 обхода в коде)

Флаг `$FE08` проверяется в **22 местах** по всему ROM. Когда `$FE08 ≠ 0`:

| Эффект | Адрес ROM | Строка в .c |
|---|---|---|
| 💧 **Вода не действует** (Labyrinth Zone) | `$3E32` | 7981, 8132 |
| 💍 **Кольца не теряются** при ударе | `$1B098` | 19688, 21122, 27327, 27471, 27502 |
| 📷 **Камера не блокируется** у боссов | `$4274` | 8425 |
| 👾 **Враги не активируются** | `$11AF6` | 17397 |
| 🎯 **Проверка попаданий отключена** | `$11DE2` | 17511 |
| 📜 **Скроллинг уровня не ограничен** | `$7B48` | 12055, 12082 |
| 🔲 **Объекты не загружаются** (Object Loader) | `$A4CE` | 13164 |
| 🏗️ **Боссы не триггерятся** | `$16EAE` | 24585 |
| 🔁 **Зонный скроллинг отключён** (5 зон) | `$FB20` | 13583, 15113, 15307, 15491, 15664 |

> **Итого:** Debug Mode = полная неуязвимость + свободное перемещение + отключение всех триггеров.

### 1.6 Warm Boot Detection

| | |
|---|---|
| **Адрес ROM** | `$000310` |
| **Файл** | sonic1_pseudo_decompilation.c, строка 694 |
| **Функция** | `EntryPoint` |

```c
/* 000310 */      compare($fffc.w, #$696e6974);  // ASCII "init"
/* 000318 */      if (Z == 1) goto loc_000364;    // совпало → тёплый старт
```

Магическое значение `$696E6974` = `"init"` записывается в RAM `$FFFC` после первой инициализации. При **Reset без выключения питания** (тёплый старт) это значение сохраняется в RAM, и игра **пропускает полную инициализацию** — быстрый рестарт с сохранением некоторых настроек.

### 1.8 Система Chaos Emeralds

|---|---|

```c
/* 01B694 */      compare($fe57.w, #$6);      // уже 6 изумрудов?
```

### 1.9 Control Lock (блокировка управления)

| | |
|---|---|
| **Адрес ROM** | `$005370–$0053E2` |
| **Файл** | sonic1_pseudo_decompilation.c, строки 8678–8710 |

```c
/* 005376 */      compare($d008.w, #$90);     // Sonic X < $90?
/* 005382 */      $f7cc.w = #$1;              // ═══ ЗАБЛОКИРОВАТЬ УПРАВЛЕНИЕ ═══
/* 005388 */      $f602.w = #$800;            // принудительно: ВПРАВО
// ... после табло очков:
```

## 2. Баги и эксплойты

### 2.1 Spike Bug — смерть через неуязвимость (КРИТИЧЕСКИЙ)

| | |
|---|---|
| **Адрес ROM** | `$01AFE0` |
| **Строка** | 27273–27280 |
| **Серьёзность** | 🔴 **КРИТИЧЕСКИЙ** |

```c
// Путь обработки урона от шипов:
loc_01AFE0:
/* 01AFE0 */      test($fe2d.w);              // есть щит?
/* 01AFE4 */      if (Z == 1) goto loc_01AFEA; // нет щита → проверить дальше
loc_01AFE6:
/* 01AFE6 */      d0 = #$ff;
/* 01AFE8 */      return;                      // с щитом → безопасно
loc_01AFEA:
/* 01AFEA */      // nop                       // ← ЗДЕСЬ ДОЛЖНА БЫТЬ ПРОВЕРКА!
/* 01AFEC */      test($30(a0));               // таймер неуязвимости
/* 01AFF0 */      if (Z == 0) goto loc_01AFE6; // таймер > 0 → безопасно
/* 01AFF2 */      a2 = a1;
  Sonic_LoseRings:                              // ПОЛУЧИТЬ УРОН
```

**Суть бага:** Когда Соник получает удар от шипов, он входит в состояние `$24 = 4` (hurt recoil). Таймер неуязвимости `$30(a0)` устанавливается в `$78` (120 кадров). Но если Соник **отскакивает обратно в те же шипы**, код повторно проходит `loc_01AFE0`, и на этот раз `$30(a0)` может быть уже 0 (или проверка обходится из-за повторного вызова `TouchResponse`).

**Сравнение с обычными врагами:**
```c
// Обычный враг (loc_01AF30):
/* 01AF36 */      compare($1c(a0), #$2);     // Sonic в Rolling?
/* 01AF3C */      if (Z == 0) goto loc_01AFE0; // нет → урон
```
Враги проверяют rolling-статус. Шипы **пропускают эту проверку** и идут напрямую к `loc_01AFE0`.

**NOP на `$01AFEA`** — место, где *должна* быть проверка `$24(a0) == 4` (hurt state). Её отсутствие — причина бага.

**Воспроизведение:** Green Hill Zone Act 3 — мост с шипами, или Marble Zone — ямы с шипами. Попасть на шипы, отскочить обратно в них → **смерть даже с кольцами**.

### 2.2 Wrap-Around Bug — X Underflow (КРИТИЧЕСКИЙ)

| | |
|---|---|
| **Адрес ROM** | `$013304` |
| **Строка** | 19601–19633 |
| **Серьёзность** | 🔴 **КРИТИЧЕСКИЙ** |

```c
/* 013304 */      d1 = $8(a0);              // X позиция Sonic (32-bit)
/* 013308 */      d0 = $10(a0);             // X скорость
/* 01330C */      d0 = sign_extend(d0);     // расширить до 32 бит
/* 01330E */      d0 <<= #$8;
/* 013310 */      d1 += d0;                 // предсказанная X позиция
/* 013312 */      d1 = swap_words(d1);      // пиксельная часть
/* 013314 */      d0 = $f728.w;             // левая граница уровня
/* 013318 */      d0 += #$10;
/* 01331C */      compare(d0, d1);          // ═══ UNSIGNED сравнение ═══
/* 01331E */      if (unsigned >) goto loc_01336C;  // → ограничить позицию

  loc_01336C:
/* 01336C */      $8(a0) = d0;              // clamp X к границе
/* 013370 */      $a(a0) = #$0;
/* 013376 */      $10(a0) = #$0;            // обнулить скорость
```

**Суть бага:** Сравнение `compare(d0, d1)` — **UNSIGNED**. Если X Sonic уходит ниже 0, значение заворачивается в `$FFxx` (очень большое беззнаковое число). Unsigned-сравнение видит это как «далеко справа от границы», и clamp **не срабатывает**.

**Результат:** Соник телепортируется на правую сторону карты уровня, часто за пределы валидных данных.

**Воспроизведение:** Бежать влево на максимальной скорости в левый край любого уровня (особенно на склоне). Координата X проходит через 0 → wrap на `$FFxx`.

### 2.4 Score Overflow — система экстра-жизней

| | |
|---|---|
| **Адрес ROM** | `$01C68C` |
| **Строка** | 27393–27405 |
| **Серьёзность** | 🟢 НИЗКИЙ |

### 2.5 Ring Counter — переполнение дисплея

| | |
|---|---|
| **Адрес ROM** | `$009C76` |
| **Строка** | 13115–13131 |
| **Серьёзность** | 🟡 СРЕДНИЙ |

```c
/* 009C76 */      $fe20.w += #$1;               // кольца += 1 (WORD, 16 бит)
```

### 2.6 Object Slot Exhaustion — пропадающие объекты

| | |
|---|---|
| **Адрес ROM** | `$00DA8C` |
| **Строка** | 14563 |
| **Серьёзность** | 🟡 СРЕДНИЙ |

```c

/* 00DA8C */      a1 = &$d800.w;               // начало пула объектов
/* 00DA90 */      d0 = #$5f;                   // 96 слотов
  loc_00DA94:
/* 00DA94 */      test((a1));                   // слот свободен?
```

### 2.13 Y Velocity — бесконечное ускорение при падении

| | |
|---|---|
| **Адрес ROM** | `$00D5C8` |
| **Строка** | 13933 |
| **Серьёзность** | 🟢 НИЗКИЙ |

```c

/* 00D5C8 */      $12(a0) += #$38;             // гравитация — БЕЗ CLAMP!
```

Гравитация добавляет `$38` к Y-скорости **каждый кадр**. Горизонтальная скорость ограничена через `$F760` (макс = `$600`). Вертикальная — **нет лимита**.

- Высокая скорость → Sonic проходит сквозь полы (floor detection работает по дискретным позициям)

## 4. Скрытые текстовые строки в ROM

### 4.1 ROM Header текстовые строки

| Смещение ROM | Содержание |
|---|---|
| `$100–$10F` | `SEGA MEGA DRIVE` (16 байт, ASCII) |
| `$110–$11F` | `(C)SEGA 1991.APR` (16 байт) |
| `$120–$14F` | `SONIC THE               HEDGEHOG` (48 байт, пробелами выровнено) |
| `$150–$17F` | `SONIC THE               HEDGEHOG` (дублируется для Overseas) |
| `$180–$18D` | `GM 00001009-00` (серийный номер) |

Обе строки (Domestic / Overseas) **идентичны** — в японской версии могло быть кана-название, но в релизе оставлен латинский дубликат.

### 4.2 Магическая строка "init"

```asm
; $00035C
000310  0C B8 69 6E 69 74 FF FC   cmpi.l  #$696e6974, $fffc.w  ; "init"
00035C  21 FC 69 6E 69 74 FF FC   move.l  #$696e6974, $fffc.w
```

`$696E6974` = ASCII `"init"`. Единственная строка, используемая как магический маркер тёплого старта. Других скрытых ASCII-строк (типа "MAJIN" или developer credits) **в коде данного ROM не обнаружено**.

### 4.3 Отсутствие пасхалки Majin / скрытых credits

**Результат:** В дампе Sonic 1 (USA/Europe) **нет** пасхалки "Majin" и скрытых credits разработчиков. Эта пасхалка присутствует **только в японской версии ROM** (другой серийный номер). В данном ROM (`GM 00001009-00`) нет:
- Никаких скрытых ASCII-строк помимо ROM-заголовка и "init"
- Нет проверок на специальные комбинации кнопок для показа credits
- `ArtNem_EndingArt` ($06270F, 16 байт) — это сжатые Nemesis-тайлы концовки, а не текстовые данные

## 5. Система определения региона

### 5.1 Чтение IO_VERSION

```asm
; Address: $00034E — внутри EntryPoint
00034E  10 39 00 A1 00 01   move.b  $a10001.l, d0  ; IO_VERSION
000354  02 00 00 C0           andi.b  #$c0, d0       ; маска: биты 7-6
000358  11 C0 FF F8           move.b  d0, $fff8.w     ; сохранить в RAM
```

**Регистр `$A10001` (IO_VERSION):**
- Бит 7 = 0: Domestic (JP), 1: Overseas (US/EU)
- Бит 6 = 0: NTSC (60Hz), 1: PAL (50Hz)
- Результат в `$FFF8`: `$00` = JP/NTSC, `$40` = JP/PAL, `$80` = US/NTSC, `$C0` = EU/PAL

**Применение:** В данном ROM значение `$FFF8` **считывается, но нигде не используется для ветвления логики**. Sonic 1 не имеет региональных различий в игровом процессе — этот код просто сохраняет значение для совместимости, вероятно для planned но не реализованной региональной логики.

### 5.2 IO_CTRL_EXT

```asm
000306  08 39 00 06 00 A1 00 0D   btst.b  #$6, $a1000d.l  ; IO_CTRL_EXT
00030E  67 0C                     beq.b   loc_00031C       ; = 0 → cold boot
```

Бит 6 `$A1000D` используется как **показатель расширения**. Если бит = 0 → не проходить проверку warm boot (всегда cold). Это дополнительная аппаратная проверка: некоторые клоны MD не устанавливают этот бит корректно.

```c
// Address: $011010, Obj_PalCycleStep

/* 01101A */      (a1) += #$1;              // инкремент счётчика
```

## 7. Special Stage: анализ макетов

### 7.1 Структура данных SS

```asm
; SS_LayoutRender ($01B14E)
01B15C  10 38 F7 80    move.b  $f780.w, d0     ; текущий угол вращения
01B160  02 00 00 FC    andi.b  #$fc, d0        ; выравнивание по 4
01B164  jsr CalcSine                            ; sin/cos для вращения
01B17C  34 38 F7 00    move.w  $f700.w, d2     ; SS X position
01B18E  36 38 F7 04    move.w  $f704.w, d3     ; SS Y position
```

Макет Special Stage хранится в RAM `$FF0000` как **сетка 128x64 байт** ($80 * $40 = $2000). Каждый байт = тип клетки:
- `$00` = пусто
- `$28` = кольцо (ring)
- `$3A` = кольцо (другой тип — GoalPost)
- `$3B–$40` = Chaos Emerald (6 типов, по одному на изумруд)
- `$41` = стена типа 1 (reverser)
- `$4A` = стена типа 2 (вместе с $41 — пара «ворот»)
- `$4E` = максимальный валидный ID клетки

```asm
; SS_LayoutRender, строка 30100
01B216  0C 00 00 4E   cmpi.b  #$4e, d0        ; ID > $4E?
01B21A  62 4C         bhi.b   loc_01B268       ; → пропустить (пустая клетка)
```

**Баг:** ID клеток `$42–$4D` (между стенами и максимумом) **не имеют назначения** — это «мёртвые» ID, которые **рендерятся** (проходят проверку `$4E`), но в SS_Results **не обрабатываются**, создавая невидимые препятствия.

### 7.2 Emerald Collection Logic

```c
// SS_Results, $01BE0C
/* 01BE0C */      compare($fe57.w, #$6);    // уже 6 изумрудов?
/* 01BE12 */      if (Z == 1) goto loc_01BE2A; // → пропустить (уже все)
/* 01BE14 */      d4 -= #$3b;               // ID изумруда = клетка - $3B
/* 01BE1A */      d0 = $fe57.w;             // текущий счёт
/* 01BE22 */      // move.b d4, ($fe58 + d0)  // записать ID в массив
/* 01BE26 */      $fe57.w += #$1;           // увеличить счётчик
```

**Находка:** Нет проверки на дубликаты при сборе изумрудов! Защита от дубликатов реализована **на уровне макета** (каждый макет содержит уникальный изумруд). Если бы можно было дважды попасть на одну клетку с изумрудом, он бы собрался дважды. Однако код `SS_ChkBlock` (вызываемый перед сбором) заменяет собранную клетку на пустую, предотвращая повторный сбор.

### 7.3 SS Reverser Gates ($41 / $4A)

```c
// SS_Results, $01BE38
/* 01BE38 */      compare(d4, #$41);        // тип = reverser?
/* 01BE3E */      $3a(a0) = #$1;            // установить флаг "gate 1 active"

/* 01BE44 */      compare(d4, #$4a);        // тип = exit gate?
/* 01BE4A */      compare($3a(a0), #$1);    // gate 1 был активирован?
/* 01BE52 */      $3a(a0) = #$2;            // обе ворота пройдены!
```

Когда `$3A(a0) == 2` (обе ворота пройдены), срабатывает преобразование макета:

```c
// SS_Results, $01BE64
/* 01BE64 */      a1 = &$ff1020.l;          // начало макета + offset
/* 01BE6E */      compare((a1), #$41);      // найти все $41
/* 01BE74 */      (a1) = #$2c;              // заменить на $2C (пустое)
```

Все ворота типа `$41` в макете **массово заменяются** на `$2C`, открывая новые пути. Это скрытый механизм «ключ-замок» в Special Stage.

## 8. Скрытые механизмы начисления очков

### 8.1 Enemy Chain Bonus — полная таблица

```c
// TouchResponse, $01AF72
/* 01AF72 */      d0 = $f7d0.w;             // chain counter
/* 01AF76 */      $f7d0.w += #$2;           // += 2 за каждого врага
/* 01AF7A */      compare(d0, #$6);         // clamp index to 6
/* 01AF80 */      d0 = #$6;                 // макс индекс таблицы
```

Таблица начисления очков при серийном уничтожении врагов (без касания земли):

| Chain # | `$f7d0` | Индекс | Очки | Спрайт |
|---|---|---|---|---|
| 1 | 0 | 0 | 100 | "100" |
| 2 | 2 | 2 | 200 | "200" |
| 3 | 4 | 4 | 500 | "500" |
| 4+ | 6+ | 6 | 1000 | "1000" |

```c
// $01AF8A — переход на 10000 очков
/* 01AF8A */      compare($f7d0.w, #$20);   // counter >= $20 (16 врагов)?
/* 01AF92 */      d0 = #$3e8;               // 1000 очков (BCD)
/* 01AF96 */      $3e(a1) = #$a;            // спрайт "10000"
```

**Скрытый бонус:** После **16 врагов** без касания земли (`$f7d0 >= $20`), каждый последующий враг даёт **10,000 очков**. Но `$f7d0` **сбрасывается** при касании пола (через `Sonic_ResetOnFloor`), не при получении урона. Это значит: можно набирать 10000-цепочки, пружиня между врагами без касания земли.

### 8.2 Monitor Bump Score — скрытая механика

```c
// TouchResponse, $01AEE4
/* 01AEE4 */      compare($30(a0), #$5a);   // timer invincibility > $5A (90)?
/* 01AEEA */      if (unsigned >=) goto loc_01AEF2;
/* 01AEEE */      $24(a1) += #$2;           // → bump monitor (дать бонус)
```

При ударе мониторов (тип `$40`), если таймер неуязвимости `$30(a0)` < `$5A` (90 кадров, т.е. первые 30 кадров после получения урона), монитор **активируется снизу**. Но если таймер >= $5A (начальные 1.5 секунды), удар снизу **не ломает монитор**. Это означает: можно разбить монитор **в первые 0.5 секунды** после получения урона, пока мерцание Sonic самое интенсивное.

### 8.3 Специальный звук урона от шипов vs. врагов

```c
// Sonic_LoseRings, $01B076
/* 01B076 */      compare((a2), #$36);      // объект $36 = spike/spikeball?
/* 01B07A */      if (Z == 0) goto loc_01B086;
/* 01B07C */      compare((a2), #$16);      // объект $16 = trap?
/* 01B082 */      d0 = #$a6;               // → СПЕЦИАЛЬНЫЙ звук!
// По умолчанию d0 = #$a3 (стандартный звук урона)
```

Объекты `$36` (шипы) и `$16` (ловушки) воспроизводят **другой звук** (`$A6`) при ударе, по сравнению с обычным `$A3`. Это различие незаметно для большинства игроков.

## 9. Неиспользуемые Level ID и граничные случаи

### 9.1 Game Mode Dispatch — 8 режимов

```asm
; $000388 — GameMode Switch
000388  10 38 F6 00    move.b  $f600.w, d0     ; текущий game mode
00038C  02 40 00 1C    andi.w  #$1c, d0        ; маска: биты 4-2 (8 значений)
000390  4E BB 00 04    jsr     $396(pc, d0.w)  ; jump table
```

Маска `$1C` допускает значения: `$00, $04, $08, $0C, $10, $14, $18, $1C`. Каждое значение — 4-байтовая инструкция `bra.w` в таблице переходов.

| `$F600` | Режим | Описание |
|---|---|---|
| `$00` | SEGA Logo | Логотип SEGA |
| `$04` | Title Screen | Титульный экран |
| `$08` | Level Play | Игровой процесс |
| `$0C` | Special Stage Exit | Выход из SS |
| `$10` | Special Stage | Специальный уровень |
| `$14` | Continue | Экран Continue |
| `$18` | Ending | Концовка |
| `$1C` | Credits | Титры |

**Находка:** Режимы `$18` (Ending) и `$1C` (Credits) **никогда не устанавливаются из обычного игрового процесса** через Level Select или нормальную игру. Они активируются только через внутреннюю логику завершения FBZ/SBZ. Однако через запись `#$18` или `#$1C` в `$F600` (например, через debug write) можно вызвать концовку/credits с любого момента.

### 9.2 Level Select: 21 запись vs реальные уровни

```c
// LevelSelect, $003460
/* 0034AA */      compare($ff82.w, #$14);   // макс. позиция курсора = 20
```

Level Select позволяет выбрать 21 запись (0–20). Реальные зоны:

| Запись | ID | Зона |
|---|---|---|
| 0–2 | $0000–$0002 | Green Hill Act 1-3 |
| 3–5 | $0100–$0102 | Marble Act 1-3 |
| 6–8 | $0200–$0202 | Spring Yard Act 1-3 |
| 9–11 | $0300–$0302 | Labyrinth Act 1-3 |
| 12–14 | $0400–$0402 | Star Light Act 1-3 |
| 15–17 | $0500–$0502 | Scrap Brain Act 1-3 |
| 18 | $0600 | Final Zone |
| 19 | — | Special Stage |
| 20 | — | Sound Test |

**Находка:** Zone ID `$0501` (Scrap Brain Act 2) внутренне перенаправляется на **Labyrinth Zone Act 4** (`$0103`):

```c
// Sonic_SlopeResist, $013346
/* 013346 */      compare($fe10.w, #$501);  // SBZ Act 2?
/* 013364 */      $fe10.w = #$103;          // → LZ Act 4!
```

Act 4 Labyrinth Zone **физически существует на карте**, но недоступен из Level Select (максимальный act для LZ = 2, т.е. Act 3). Scrap Brain Act 3 = LZ Act 4 с палитрой SBZ.

### 9.3 Нет Zone $07+ — что происходит?

Zone ID хранится в `$FE10` (word: старший байт = зона, младший = акт). Код загрузки размеров уровня:

```c
// LevelSizeLoad, $005ED4
/* 005ED4 */      d0 = $fe10.w;
/* 005ED8 */      d0 <<= #$6;               // × 64
/* 005EDA */      d0 <<= #$4;               // ещё × 16 = × 1024 total
// используется как индекс для PC-relative таблицы
```

Если zone ID > 6 (Final Zone), `d0 = zone_id * 1024` выходит за пределы таблицы размеров уровней. Результат: **чтение мусорных данных** → неопределённые границы уровня, крэш или чёрный экран.

## 10. Обход ограничения скорости

### 10.1 Обычный Speed Cap

```c
// Sonic_Move, $012EB0
// Загрузка параметров:
/* 012EB4 */      $f760.w = #$600;          // макс. скорость (на суше)
/* 012EBA */      $f762.w = #$c;            // ускорение
/* 012EC0 */      $f764.w = #$80;           // замедление

// В воде:
/* 012EC6 */      $f760.w = #$300;          // вдвое медленнее
/* 012ECC */      $f762.w = #$6;
/* 012ED2 */      $f764.w = #$40;
```

Speed cap `$600` = 1536 субпикселей/кадр ≈ 6 пикселей/кадр.

### 10.2 Способ 1: Spring Launch (обход через Y→X конверсию)

Когда Sonic приземляется на наклонную поверхность, Y-скорость конвертируется в X (ground speed) через `Sonic_CheckFloor`:

```c
// Sonic_CheckFloor, $01340C
/* 01340C */      d2 = #$680;               // ═══ LAUNCH SPEED = $680 ═══
/* 013418 */      d2 = #$380;               // в воде = $380
/* 01342C */      d1 *= d2;                 // CalcSine(angle) * speed
/* 013434 */      d0 *= d2;
/* 013438 */      $12(a0) += d0;            // add to Y velocity
```

Значение `$680` > `$600` (speed cap). При пружине, бросающей Sonic на наклонную поверхность, его горизонтальная скорость **превышает speed cap** через sin/cos проекцию. Speed cap из `Sonic_Move` применяется **только при нажатии D-pad**, а не при инерционном движении.

### 10.3 Способ 2: Slope Speed Uncapped Rolling

```c
// Sonic_RollSpeed (не показан напрямую, но работает так):
// При rolling (animation #$2), slope resistance:
/* 01329C */      $f73e.w += #$4;           // camera look-ahead увеличивается
/* 0132A0 */      $f73e.w -= #$2;
```

Когда Sonic катится вниз по склону, гравитация (`ObjectFall: += $38 каждый кадр`) добавляет к ground speed через проекцию угла. Speed cap в `Sonic_MoveLeft`/`Sonic_MoveRight` **не применяется**, т.к. эти функции проверяют направление D-pad, а при инерционном скольжении вниз по склону Sonic ускоряется **без ограничений**.

### 10.4 Способ 3: Water Exit Double Speed

```c
// Sonic_Water, при выходе из воды:
/* 012DDE */      $12(a0) = $12(a0) * 2;   // Y velocity × 2
/* 012DE2 */      compare($12(a0), #$f000);
/* 012DE8 */      $12(a0) = #$f000;         // clamp to -$1000
```

При выходе из воды Y-скорость **удваивается** с clamp до `$F000` (-4096). Это значительно выше нормального speed cap `$600`. Если Sonic выходит из воды при движении вверх по наклонной, эта скорость конвертируется в ground speed через `Sonic_CheckFloor`, создавая **сверхзвуковое ускорение**.

## 11. Баг персистентности звёзд неуязвимости

### 11.1 Механизм Invincibility Timer

```c
// Sonic_InvincTimer, $0120C0
/* 0120C0 */      test($36(a0));             // таймер != 0?
/* 0120C6 */      test_bit($22(a0), #$3);   // Sonic стоит на объекте?
/* 0120CE */      $36(a0) = #$1e;           // если да → сбросить таймер на 30

/* 0120D6 */      $36(a0) -= #$1;           // декремент
/* 0120DC */      $28(a0) += #$1;           // при 0: следующий субрутин
/* 0120E0 */      $38(a0) = 0;              // очистить вспомогательный таймер
```

**Баг:** Когда Sonic стоит на движущемся объекте (бит 3 `$22(a0)` = 1), таймер неуязвимости **постоянно сбрасывается** на `$1E` (30 кадров). Пока Sonic стоит на платформе, неуязвимость **никогда не истечёт**. Звёзды неуязвимости продолжают отображаться бесконечно.

### 11.2 Transition Persistence

При переходе между актами объект Sonic **не пересоздаётся**. Поля `$30(a0)` (invincibility flash timer) и `$36(a0)` (invincibility star timer) **переносятся** в следующий акт. Если Sonic входит в переход с активной неуязвимостью, мерцание продолжится в следующем акте до естественного истечения таймера. Нет явного кода очистки `$30`/`$36` при загрузке нового акта.

## 12. Секреты циклической палитры

### 12.1 PalCycle_Underwater — подводная палитра

```c
// PalCycle_Underwater, $00195C
/* 001970 */      $f634.w = #$5;            // таймер = 5 кадров
/* 001976 */      d0 = $f632.w;             // offset в палитре
/* 00197A */      $f632.w += #$1;           // следующий кадр
/* 00197E */      d0 &= #$3;               // 4 состояния циклично (0-3)
/* 001982 */      d0 <<= #$3;              // × 8 байт = offset в таблице
/* 001984 */      a1 = &$fb50.w;           // целевой адрес палитры
```

Подводная палитра циклирует **4 цвета** каждые **6 кадров** (5 кадров ожидания + 1 обновление = ~10Hz). Данные палитры берутся из двух таблиц:
- `$1B66` — основная подводная палитра
- `$1B86` — альтернативная (для другого акта)

**Скрытая деталь:** функция имеет **два входа** — `$00195C` (основная) и `$001964` (альтернативная). Второй вход загружает палитру из `$1B86`. Какой вход используется, зависит от вызывающего кода. Это означает, что LZ может иметь **два разных визуальных стиля воды** в зависимости от акта.

### 12.2 PaletteFadeIn — алгоритм постепенного проявления

```c
// PaletteFadeIn, $001DAC
/* 001DCA */      d4 = #$15;               // ═══ 21 шаг затухания ═══
// Каждый шаг:
/* 001E30 */      d1 = d3 + #$200;         // +2 к красному (бит 9-11)
/* 001E3E */      d1 = d3 + #$20;          // +2 к зелёному (бит 4-6)
/* 001E4A */      (a0)+ += #$2;            // +2 к синему (бит 1-3)
```

Палитра проявляется **поканально**: сначала красный (+$200), затем зелёный (+$20), затем синий (+$2). Всего 21 шаг. SEGA MD использует формат BGR: 3 бита на канал (9 бит цвета, 512 цветов). Каждый канал увеличивается на 2 за шаг → 21 шаг × 3 канала = полный fade от чёрного до целевого цвета.

**Специальная обработка LZ:** При `$FE10 == 1` (Labyrinth Zone) дополнительно fadeдится **подводная палитра** из `$FA00`/`$FA80`:

```c
/* 001E00 */      compare($fe10.w, #$1);    // Labyrinth Zone?
/* 001E0A */      a0 = &$fa80.w;           // underwater palette target
/* 001E0E */      a1 = &$fa00.w;           // underwater palette source
```

### 13.1 Таблица анимаций Sonic

```c
// Sonic_Animate, animation table at $13B48
// Каждая запись = последовательность frame ID, завершающаяся $FF/$FE/$FD
```

Основные состояния анимации (`$1C(a0)`):

| ID | Анимация | Используется? |
|---|---|---|
| $00 | Walk | ✅ |
| $01 | Run | ✅ |
| $02 | Rolling | ✅ |
| $03 | Roll (init jump) | ✅ |
| $04 | Push | ✅ |
| $05 | Wait (idle) | ✅ |
| $06 | Balance (edge 1) | ✅ |
| $07 | Look up | ✅ |
| $08 | Crouch | ✅ |

| $0A | Skid/Brake | ✅ |

| $0D | Brake (ground) | ✅ |
| $18 | Death | ✅ |
| $1A | Hurt | ✅ |
| $1B | Waterslide | ✅ |

### 13.2 Порог Walk → Run

```c
// Sonic_Animate, $13AA2
/* 013AA2 */      compare(d0, #$600);       // ground speed >= $600?
// → При $600+: переключение с Walk frames ($13B86) на Run frames ($13B8E)
```

Переключение Walk→Run происходит **ровно** при speed cap `$600`. Это значит: при нормальном беге Sonic **никогда не достигает** анимации Run, т.к. speed cap = `$600`, и нужно `>= $600`. Анимация Run видна **только при ускорении через склон или Speed Shoes** (когда cap повышается).

### 13.3 Rolling Animation Speed Threshold

```c
// Sonic_Animate, $13ADE
/* 013ADE */      compare(d0, #$600);       // тот же порог для rolling
```

Тот же порог `$600` определяет, какой набор кадров rolling используется. Два набора кадров rolling отличаются скоростью смены — быстрый rolling при высокой скорости выглядит как «размытие».

## 14. Границы Jump-таблиц и потенциальные OOB

### 14.1 Object Dispatch Table

```asm
; ExecuteObjects, $00D350
00D34C  D0 40          add.w  d0, d0          ; ID × 2
00D34E  D0 40          add.w  d0, d0          ; ID × 4
00D350  movea.l $d37e(pc, d0.w), a1           ; read pointer from table
00D354  jsr    (a1)                            ; CALL
```

Object ID из байта 0 объектного слота. **Нет bound check.** Если object ID > максимального → чтение за пределами таблицы → джамп по произвольному адресу.

Таблица объектов начинается с `$D37E` (PC-relative). Максимальный валидный ID = количество записей в таблице / 4. При ID выходящем за пределы, `d0.w` индексирует данные **после таблицы** (функции ObjectFall, SpeedToPos, DisplaySprite) — они будут интерпретированы как адреса → непредсказуемое поведение.

### 14.2 Boss Routine Dispatch

```c
// Obj3D_GHZBoss_Chain, $017EE4
/* 017EE4 */      // move.w $17f0e(pc, d0.w), d1
/* 017EE8 */      call($17f0e(pc, d1.w));
```

Каждый босс использует **свою jump table** с PC-relative индексацией. Поле `$25(a0)` определяет текущий суб-рутин. Если `$25` выходит за пределы таблицы (возможно при memory corruption), бос выполнит произвольный код.

### 14.3 Animation Jump Table

```c
// Sonic_Animate, $13AA0
// d0 = animation ID ($1c(a0))
// Если animation ID > количества записей в таблице $13B48 → OOB read
```

Нет явной проверки верхней границы animation ID. Если `$1C(a0)` установлен в невалидное значение (например через memory corruption), система анимации прочитает данные за пределами таблицы.

## 15. Эксплойты звукового драйвера

### 15.1 Архитектура Sound Driver

```asm
; SoundDriver_Main, $071B4C
071B4C  move.w  #$100, $a11100.l  ; запросить Z80 bus
071B5A  btst.b  #$7, $a01ffd.l    ; Z80 готов?
071B6C  beq.b   loc_071B82        ; да → продолжить
071B80  bra.b   SoundDriver_Main  ; нет → retry
```

Sound Driver работает на **68000** (не на Z80 как в поздних играх SEGA). Он **блокирует Z80 bus** на время работы:
- Запрос bus: `$A11100 = $100`
- Проверка бита 7 `$A01FFD`
- Если Z80 не готов → **бесконечный цикл retry** (`bra.b SoundDriver_Main`)

**Баг:** Если Z80 зависнет (hardware fault), Sound Driver войдёт в **бесконечный цикл**, полностью замораживая игру. Нет timeout.

### 15.2 Sound Channel Architecture

```asm
; SoundDriver_Main — обновление каналов:
071BD8  moveq  #$5, d7           ; 6 FM каналов (0-5)
071BDA  adda.w #$30, a5          ; каждый канал = $30 байт
071BE2  jsr    SndDrv_UpdateFM   ; обновить FM

071BEA  moveq  #$2, d7           ; 3 PSG канала (0-2)
071BEC  adda.w #$30, a5
071BF4  jsr    SndDrv_UpdatePSG  ; обновить PSG
```

Всего: **6 FM + 3 PSG + 1 DAC = 10 каналов**. Каждый канал занимает `$30` (48) байт в RAM `$FFF000+`.

### 15.3 Sound Test Limits

```c
// LevelSelect, Sound Test
/* 00355A */      d0 = $ff84.w;             // sound test value: 0–$4F
/* 00355E */      d0 += #$80;               // → PlaySound ID: $80–$CF
```

Sound Test допускает значения `$00–$4F`, которые транслируются в IDs `$80–$CF`. Но PlaySound/PlaySound_Special **не ограничивают** ID:

```c
// PlaySound, $001396
/* 001396 */      $f601.w = d0;             // записать ID в очередь
// PlaySound_Special, $00139C
/* 00139C */      $f609.w = d0;             // записать в специальную очередь
```

IDs за пределами `$80–$CF` (например `$D0+`) обрабатываются `SndDrv_CmdHandler` как **команды**, а не музыка:
- `$E0` — Fade out
- `$E1`–`$E4` — специальные музыкальные треки
- `$E5`+ — **недокументированные команды** → чтение данных за пределами таблицы треков

## 16. Проблемы массива столкновений (FindBlock)

### 16.1 Структура Collision Array

```c
// FindBlock, $01495C
/* 01495C */      d0 = d2;                  // X координата
/* 01495E */      d0 <<= #$1;
/* 014960 */      d0 &= #$380;             // маска: $380 = 7 бит (128 блоков по X)
/* 014964 */      d1 = d3;                  // Y координата
/* 014966 */      d1 <<= #$8;
/* 014968 */      d1 &= #$7f;              // маска: $7F = 7 бит (64 блоков по Y! НЕТ — это 128)
/* 01496E */      d1 = #$ff;               // ← BUG: d1 перезаписывается на $FF!
/* 014970 */      a1 = &$a400.w;           // collision array base
/* 014974 */      // move.b (a1, d0.w), d1
```

**Важное наблюдение:** На строке `$01496E` d1 **перезаписывается** на `$FF` как значение по умолчанию (тип блока "air"). Фактически вычисление Y-компонента `d1 &= #$7f` **теряется**, т.к. сразу перезаписывается. Реальная индексация таблицы столкновений использует **только X-компонент** (d0), а d1 устанавливается из `(a1, d0.w)`.

Collision array `$A400` имеет размер `$400` (1024 байт). Маска `$380` ограничивает X-индекс до 128 блоков × 8 = 1024. **Нет OOB** — маска корректно ограничивает доступ.

### 16.2 Layer Switching

```c
// FindBlock, $01499A
/* 01499A */      d1 &= #$7f;              // высокий блок: Layer B
/* 0149A6 */      d1 += #$1;               // ← смещение +1
/* 0149A8 */      compare(d1, #$29);        // ID = $29?
/* 0149AE */      d1 = #$51;               // → подменить на $51!
```

Блок ID `$29` в Layer B **специально подменяется** на `$51` при активном бите 6 `$01(a0)` (render flag). Это **скрытый механизм**: блок `$29` действует как «портал» между слоями столкновений. Sonic переключается между передним и задним планом через этот специальный ID.

## 17. Баги взаимодействия объектов

### 17.1 Signpost Solid Object Bug

```c
// Obj3E_Signpost_Main, $01A172
/* 01A172 */      compare($34(a0), #$c);    // sub-routine >= $C?
/* 01A178 */      if (>= $C) → return       // → НЕ быть solid!
/* 01A17A */      d1 = #$1b;                // width = $1B
/* 01A17E */      d2 = #$70;                // full height = $70
/* 01A182 */      d3 = #$71;                // display height = $71
/* 01A18A */      jmp SolidObject            // → обрабатывается как SOLID
```

Signpost (табличка конца акта) является **solid объектом** (SolidObject) с шириной `$1B` (27 пикселей) и высотой `$70` (112 пикселей). Sonic может **стоять на ней**! Когда `$34(a0) >= $C` (после 6 циклов вращения), Signpost **перестаёт быть solid** и проваливается сквозь Sonic.

### 17.2 Signpost Spin — невидимый hitbox

```c
// Obj3E_Signpost_Spin, $01A1C6
/* 01A1C6 */      $20(a0) = #$f;            // collision property = $F
```

После активации Signpost получает collision property `$0F` — это делает его **уязвимым к TouchResponse** как вражеский объект. Во время вращения Signpost можно **ударить как врага** (при определённых условиях Rolling), что вызовет дополнительную обработку коллизии.

### 17.3 Prison Capsule (Obj_EndSeq) — множественные дочерние объекты

```c
// Obj_EndSeq, $01A472
/* 01A472 */      d0 = $28(a0);             // sub-type
// sub-type 0 = контейнер (lid)
// sub-type 1 = содержимое (animals)
// sub-type 2+ = частицы с bit 1 render flags
/* 01A494 */      test($28(a0));
/* 01A498 */      if ($28 > 2) → set render bit 1
```

Capsule создаёт **8 дочерних объектов-животных** (`$3F`) через `Obj_RingScatter_Move`. Каждый животный объект с random-позицией:

```c
// Obj_RingScatter_Move, $01AC84
/* 01AC84 */      RandomNumber();
/* 01AC8E */      d1 <<= #$2;              // × 4
/* 01AC90 */      d1 -= #$20;              // offset ±$20
/* 01AC94 */      $8(a1) += d1;            // случайное X смещение
/* 01AC98 */      d0 <<= #$8;
/* 01AC9A */      d0 <<= #$3;              // × 2048
/* 01AC9C */      $c(a1) += d0;            // случайное Y смещение
```

**Баг:** RandomNumber используется для позиционирования частиц, но `d0 <<= #$8; d0 <<= #$3` = сдвиг на **11 бит**. Это огромное смещение `$c(a1)` (до ±$7FF00), что может привести к отображению частиц **за пределами экрана** или даже за пределами валидного Y-пространства.

## 18. Баги перехода между актами

### 18.1 GotThroughAct — Control Lock

```c
// GotThroughAct, $005370
/* 005376 */      compare($d008.w, #$90);   // Sonic X < $90?
/* 005382 */      $f7cc.w = #$1;            // LOCK управление
/* 005388 */      $f602.w = #$800;          // FORCE ВПРАВО
```

Три фазы перехода:
1. **Фаза 0** (`$f7d7 = 0`): Sonic X < `$90` → заблокировать управление, форсировать вправо
2. **Фаза 2** (`$f7d7 = 2`): Sonic X >= `$A0` → разблокировать, остановить, выставить анимацию победы (#$505 = look right + V-sign), установить камеру автоскролл
3. **Фаза 4** (`$f7d7 = 4`): Зафиксировать Sonic на X = `$A0`, изменить Object ID на `$87`

**Баг edge case:** Если Sonic достигает X = `$90` в фазе 0, но `$d008` (X position word) из-за sub-pixel wrap уже >= `$90` на следующий кадр, фаза 0 **пропускается**. Sonic остаётся с заблокированным управлением, бегущим вправо бесконечно.

### 18.2 SBZ Act 2 → LZ Act 4 — утечка палитры

```c
// Sonic_SlopeResist, $013346
/* 013346 */      compare($fe10.w, #$501);  // Scrap Brain Act 2?
/* 01334C */      if (Z == 0) goto Sonic_LoseRings_Sub;
/* 01335A */      $fe30.w = 0;              // очистить checkpoint
/* 013364 */      $fe10.w = #$103;          // → LZ Act 4
```

При переходе SBZ2→LZ4 **не происходит**:
- Сброс `$F632` (palette cycle offset)
- Сброс `$F634` (palette cycle timer)
- Сброс `$F635` (palette sub-counter)
- Переинициализация `$FA00`–`$FA80` (underwater palette)

Результат: палитра SBZ «утекает» в LZ4 → первые несколько кадров вода в LZ4 имеет **неправильные цвета** (оранжевый/красный вместо синего), пока PalCycle_Underwater не перезапишет значения.

### 18.3 Continue_SetupLevel — скрытая инициализация

```c
// Continue_SetupLevel, $005898
/* 005898 */      d0 = $fff4.w;             // continue counter
/* 00589C */      d0 &= #$f;               // маска 0-15
/* 0058AE */      compare($fff4.w, #$9);   // 9 continues max
/* 0058DA */      compare($fff4.w, #$4);   // если 4 continues:
/* 0058E2 */      a1 = &$5908.l;           // → загрузить СПЕЦИАЛЬНЫЕ данные
/* 0058F0 */      (a2)+ = (a1)+;           // копировать 9 long-word
```

При **ровно 4 continues** (`$FFF4 == 4`), загружается специальный блок данных из `$5908` в `$FE30+`. Это **скрытое восстановление состояния** — вероятно предназначенное для внутреннего тестирования, позволяющее стартовать с определённым набором предметов и прогрессом.

## 19. Multi-Hit Boss Exploits

### 19.1 GHZ Boss — механизм урона

```c
// Obj3D_GHZBoss_Main — проверка урона:
// Boss HP хранится в $21(a1) (hit counter enemy property)
// TouchResponse, $01AF40:
/* 01AF40 */      test($21(a1));            // HP != 0?
/* 01AF46 */      $10(a0) = -$10(a0);      // отразить X скорость Sonic
/* 01AF4A */      $12(a0) = -$12(a0);      // отразить Y скорость
/* 01AF5C */      $21(a1) -= #$1;           // HP -= 1
/* 01AF62 */      if ($21 == 0) → set defeated bit
```

**Нет кадров неуязвимости для босса!** `$21(a1)` декрементируется **каждый кадр** пока Sonic находится в хитбоксе босса. Если Sonic остаётся внутри хитбокса **несколько кадров**, HP уменьшается **многократно**.

### 19.2 Multi-Hit Window

```c
// TouchResponse — проверка boss-type:
/* 01AF30 */      test($fe2d.w);            // Sonic has shield?
/* 01AF36 */      compare($1c(a0), #$2);   // Sonic в rolling? (animation #$2)
/* 01AF3C */      if (Z == 0) → HURT Sonic  // не в rolling → получить урон
```

**Условие попадания по боссу:** Sonic должен быть в rolling state (`$1C == 2`). Но после первого удара:
1. `$10(a0)` отражается (Sonic отскакивает)
2. `$12(a0)` отражается
3. НО rolling state **не сбрасывается**!

Если momentum достаточно мал (Sonic медленно касается босса), отражение скорости **не отбрасывает** его из хитбокса за один кадр → следующий кадр снова детектирует коллизию → ещё одно попадание.

### 19.3 Практическое применение

Все боссы в Sonic 1 используют одну и ту же `TouchResponse` логику:
- **GHZ Boss** (Obj3D): 8 HP → можно убить за 2-3 удара при правильном angle
- **MZ Boss** (Obj47): аналогично
- **SYZ Boss** (Obj4A): аналогично  
- **LZ Boss** (Obj4B): особый — кастомная обработка, менее подвержен multi-hit
- **SLZ Boss** (Obj4C): аналогично standard
- **SBZ Boss** (Obj4D): 8 HP, но нестандартная атака → multi-hit затруднён

**Boss Score:** `AddPoints(d0 = #$64)` = 100 BCD очков за босса. Вызывается **однократно** при defeat, не при каждом ударе.

## 20. Поведение Signpost — скрытые детали

### 20.1 Camera Unlock при Signpost

```c
// Obj3E_Signpost_Main, $01A166
/* 01A166 */      compare($f72a.w, #$2700); // правая граница камеры < $2700?
/* 01A16E */      $f72a.w += #$2;           // РАСШИРИТЬ границу на 2
```

Signpost **плавно расширяет правую границу камеры** по 2 пикселя за кадр, пока не достигнет `$2700`. Это создаёт эффект «открывания» экрана вправо после прохождения акта.

### 20.2 Signpost Spin Physics

```c
// Obj3E_Signpost_Spin, $01A192
/* 01A19A */      $2(a0) = #$400;           // VRAM pattern = $400
/* 01A1A0 */      $1c(a0) = #$0;            // animation = 0 (spin)
/* 01A1A6 */      set bit 0 of $22(a0)      // direction = RIGHT
/* 01A1AC */      jsr SpeedToPos             // применить скорость
/* 01A1B2 */      compare($c(a0), #$544);   // Y < $544?
/* 01A1BA */      $10(a0) = #$180;           // X speed = $180 (вправо)
/* 01A1C0 */      $12(a0) = #$ffe8;          // Y speed = -$18 (вверх)
```

После активации Signpost **летит вверх-вправо** со скоростью `$180` по X и `-$18` ($FFE8) по Y. Когда Y < `$544`, движение прекращается. Signpost получает collision property `$0F` и увеличивает `$34(a0)` на 2 за кадр.

## 21. Эксплойт таймера утопления

### 21.1 Механизм Drowning

```c
// Obj_DrownTimer, $019FD6
/* 019FD6 */      a1 = $36(a0);             // указатель на родительский объект
// Обратный отсчёт:
/* 019FEE */      $30(a0) -= #$1;           // декремент
/* 019FF2 */      if ($30 < 0) → reset to 0, set parent $29 = $FF (drown!)
```

### 21.2 Звук предупреждения

```c
/* 01A00A */      d0 = $fe0e.w;             // глобальный кадровый счётчик
/* 01A00E */      d0 &= #$f;               // каждые 16 кадров
/* 01A012 */      if (d0 != 0) → skip
/* 01A014 */      d0 = #$b1;               // звук "тик" утопления
/* 01A018 */      PlaySound();
```

Предупреждающий звук `$B1` играет **каждые 16 кадров** (≈3.75 раз в секунду). Это использует **глобальный счётчик** `$FE0E`, а не собственный таймер объекта.

### 21.3 Эксплойт: отмена утопления выходом из воды

Таймер утопления (`$30(a0)`) принадлежит **дочернему объекту** (пузырьки), а не Sonic. При выходе из воды `Sonic_Water` **не удаляет** объект пузырьков явно — он удаляется через `ChkObjOnScreen` когда выходит за пределы экрана.

**Эксплойт:** Если Sonic **быстро** входит и выходит из воды (например, подпрыгивая у поверхности), объект пузырьков может **не удалиться** вовремя, но таймер продолжит работать в фоне. При повторном входе в воду может быть создан **второй** объект пузырьков, и два одновременных таймера будут конфликтовать.

### 21.4 Water Slide — скрытый механизм

```c
// Sonic_WaterSlide (внутри Camera_SetLimits области)
// Проверяет collision block type для специальных "slide" блоков:
/* 3F34 */      // Reads collision array for water slide trigger blocks
// При обнаружении slide-блока:
/* 3F50 */      $14(a0) = ground_speed;    // устанавливает ground speed
/* 3F56 */      $1c(a0) = #$1b;            // animation = $1B (waterslide!)
/* 3F5A */      d0 = #$d0;                  // звук всплеска
/* 3F5E */      PlaySound();
```

Водные горки в Labyrinth Zone используют **специальные блоки столкновений** (не объекты). При контакте с таким блоком Sonic получает:
- Принудительную анимацию `$1B` (waterslide)
- Звук `$D0` (всплеск, переиспользуется)
- Ground speed вычисляется из текущей позиции на склоне

## 22. Ring Art Decompression — особенности

### 22.1 Ring Scatter Limit

```c
// Sonic_LoseRings, $01AFF4
/* 01AFFA */      test($fe20.w);            // rings != 0?
/* 01AFFE */      if (Z == 1) → goto death  // 0 колец → смерть!
/* 01B002 */      FindFreeObj();             // создать Obj $37
```

При ударе создаётся **один** объект `$37` (RingScatter), который затем порождает визуальные кольца. Но:

```c
// Obj37_RingScatter, $01ACFA
/* 01ACFA */      d0 = #$7;
/* 01ACFC */      d0 &= $fe0f.w;            // каждые 8 кадров (frame counter & 7)
/* 01AD00 */      if (d0 != 0) → skip        // создавать только каждый 8й кадр!
```

Рассыпание колец происходит **не мгновенно** — каждый 8й кадр создаётся одно визуальное кольцо через `FindFreeObj`. Если свободных объектных слотов нет, кольцо **не создаётся**. При 96 объектных слотах и большом количестве колец часть колец **визуально теряется**.

### 22.2 Ring Scatter Timer

```c
/* 01AD38 */      $1e(a0) -= #$1;           // таймер рассыпания
/* 01AD42 */      $1e(a0) = #$b4;           // сброс на 180 кадров (3 сек)
```

Рассыпанные кольца существуют **$B4** (180) кадров = 3 секунды, затем объект переходит в следующую фазу (мерцание и исчезновение).

### 22.3 Формула случайного разброса

```c
// Obj37_RingScatter, $01AD1C
/* 01AD1C */      RandomNumber();
/* 01AD22 */      d0 &= #$1f;              // 0–31
/* 01AD26 */      d0 -= #$6;               // -6 to +25 (ASYMMETRIC!)
```

Разброс колец по X: **от -6 до +25 пикселей**. Формула **асимметрична** — кольца чаще рассыпаются **вправо** от Sonic. Это видно при замедленном воспроизведении.

## 23. Дополнительные находки

### 23.1 Checksum Failure — красный экран

```asm
; EntryPoint, при ошибке checksum:
0003B8  bsr.w  VDPSetupGame
0003C8  move.w #$e, $c00000.l    ; залить CRAM цветом $E (красный)
0003D4  bra.b  $3d4              ; ═══ БЕСКОНЕЧНЫЙ ЦИКЛ ═══
```

При неверной контрольной сумме ROM (сравнение на `$18E`) экран заливается **красным** (Genesis blue = `$00E` = 14 = красный в формате BGR), и консоль **навсегда зависает**. Единственный способ выйти — отключить питание.

### 23.2 Exception Handlers — отладочная информация

```asm
; BusError ($0003D6):
0003D6  move.b #$2, $fc44.w      ; записать код ошибки

; AddressError ($0003DE):
0003DE  move.b #$4, $fc44.w

; IllegalInstruction ($0003E6):
0003E6  move.b #$6, $fc44.w
; + addq.l #$2, $2(a7)           ; пропустить инструкцию и продолжить!
```

**Скрытая отладочная система:** При exceptions записывается код ошибки в `$FC44`. IllegalInstruction **не крэшит** — оно пропускает инструкцию (`addq.l #$2, $2(a7)`) и продолжает выполнение! Это позволяет ROM работать на **неполных эмуляторах**, пропуская не-реализованные опкоды.

### 23.3 $FFFA — Invincibility Flag

```c
// Sonic_LoseRings, $01B090
/* 01B090 */      test($fffa.w);            // ═══ ГЛОБАЛЬНЫЙ ФЛАГ НЕУЯЗВИМОСТИ ═══
/* 01B094 */      if ($FFFA != 0) → DO hurt but DON'T kill
```

`$FFFA` — **недокументированный глобальный флаг**. Когда `$FFFA != 0`, вместо смерти (при 0 колец) Sonic получает урон через `Sonic_LoseRings` → `loc_01B01C` (hurt animation + отскок), но **не умирает**. Этот флаг **нигде в ROM не устанавливается** в 1. Это **вырезанная фича "бессмертия"**, доступная только через внешнюю запись в RAM.

### 23.4 Object $87 — Post-Level Sonic

```c
// GotThroughAct, $0053D6
/* 0053D6 */      $d000.w = #$87;           // заменить Object ID Sonic на $87!
```

После прохождения акта Object ID Sonic **заменяется с обычного ($01) на $87**. Объект `$87` — это **"стоящий Sonic" без AI**: он рендерится, но не обрабатывает физику, управление, и столкновения. Это переключение из «игрового персонажа» в «элемент кат-сцены».

## 24. ROM Header и таблица векторов прерываний

### 24.1 Таблица векторов (0x000000–0x0000FF)

Первые 256 байт ROM — стандартная таблица Motorola 68000 exception vectors:

```asm
; Адрес $000000 — Initial SSP (Stack Pointer)
000000  00 FF FE 00    ; SSP = $00FFFE00 (вершина RAM)

; Адрес $000004 — Entry Point (Reset vector)
000004  00 00 02 06    ; PC = $00000206 → EntryPoint

; Адрес $000068 — Level 4 Interrupt (HBlank)
000068  00 00 11 26    ; HBlank_Handler = $001126

; Адрес $000078 — Level 6 Interrupt (VBlank)
000078  00 00 0B 10    ; VBlank_Handler = $000B10
```

**Ключевые детали:**
- **SSP = `$FFFE00`** — стек растёт вниз от вершины 64КБ RAM (`$FF0000–$FFFFFF`)
- **EntryPoint = `$000206`** — не `$000000`, первые 512 байт — заголовок + векторы
- **HBlank = `$001126`** — обработчик горизонтального прерывания (для эффекта воды в LZ)
- **VBlank = `$000B10`** — главный обработчик вертикального прерывания (DMA, ввод, звук)
- Все неиспользуемые векторы (Bus Error, Address Error и т.д.) направлены на обработчики ошибок в `$0003D6+`

### 24.2 ROM Header ($000100–$0001FF)

```asm
; $100: "SEGA MEGA DRIVE "  (16 байт, пробелами до 16)
; $110: "(C)SEGA 1991.APR"  (дата: апрель 1991)
; $120: "SONIC THE               HEDGEHOG"  (48 байт, Domestic)
; $150: "SONIC THE               HEDGEHOG"  (48 байт, Overseas — идентично)
; $180: "GM 00001009-00"     (серийный номер)
; $18E: XX XX                 (контрольная сумма ROM)
; $1A0: 00000000 0007FFFF    (ROM: 0–$7FFFF = 512 КБ)
; $1A8: 00FF0000 00FFFFFF    (RAM: $FF0000–$FFFFFF = 64 КБ)
```

### 24.3 Таблица функций (580 функций)

## 25. Верификация контрольной суммы и красный экран смерти

### 25.1 Алгоритм подсчёта ($00031C–$0003D4)

```asm
; $00031C — Checksum computation
00031C  41 F8 02 00               lea.l    $200.w, a0        ; начало = $200 (после заголовка)
000320  30 39 00 00 01 A4         move.w   $1a4.l, d0        ; конец ROM из заголовка
000328  E0 58                     ror.w    #$8, d0           ; swap bytes (big-endian fix?)
00032A  72 00                     moveq    #$0, d1           ; аккумулятор = 0

; Цикл суммирования:
00032C  D2 58                     add.w    (a0)+, d1         ; d1 += *a0++ (word)
00032E  B1 C8                     cmpa.l   a0, a0            ; ...
000332  ; ... loop until a0 reaches ROM end

; Сравнение с эталоном:
000338  23 C8 00 00 01 8E         movea.l  #$18e, a1         ; адрес эталонной суммы
00033E  B2 51                     cmp.w    (a1), d1          ; computed == stored?
000340  64 76                     bcc.b    loc_0003B8        ; MATCH → нормальная инициализация
```

Алгоритм: **сумма всех 16-битных слов** ROM от `$200` до конца. Результат сравнивается с word по адресу `$18E` в заголовке.

### 25.2 Нормальный путь (контрольная сумма совпала)

```asm
; loc_0003B8 → CheckSumOk path
; Нормальная инициализация:
000342  bsr.w    VDPSetupGame        ; настройка VDP регистров
000348  bsr.w    SoundDriverLoad     ; загрузка звукового драйвера в Z80
00034E  ; ... очистка RAM, инициализация джойпадов

; Главный цикл:
000388  move.b   $f600.w, d0         ; текущий Game Mode
00038C  andi.w   #$1c, d0            ; маска: 8 режимов
000390  jsr      $396(pc, d0.w)      ; jump table → SEGA/Title/Level/SS...
000394  bra.b    $388                ; бесконечный цикл
```

### 25.3 Путь ошибки — красный экран ($0003B8–$0003D4)

```asm
; CHECKSUM FAILED:
0003B8  bsr.w    VDPSetupGame        ; минимальная настройка VDP
0003C8  move.w   #$e, $c00000.l      ; ═══ CRAM = $000E (красный) ═══

; БЕСКОНЕЧНЫЙ ЦИКЛ:
0003D4  bra.b    $3d4                ; ← ВЕЧНЫЙ HANG
```

**Цвет `$000E`** в формате Genesis BGR (0BBB-0GGG-0RRR):
- B = 0, G = 0, R = $E = 14 → **максимальный красный**
- Весь экран заливается одним цветом через запись в CRAM
- Консоль **навсегда зависает** — нет выхода кроме отключения питания

**Практическое значение:** Любая модификация ROM (хаки, переводы) должна пересчитать контрольную сумму по адресу `$18E`, иначе — красный экран. Многие ранние ROM-хаки 90-х не делали этого, что приводило к зависанию на реальном железе.

## 26. VDP-трюк: HBlank-прерывание и палитра воды

### 26.1 Обработчик HBlank ($001126)

```asm
; HBlank_Handler — вызывается при каждом HBlank
001126  48 E7 FF FE               movem.l  d0-d7/a0-a6, -(a7)  ; сохранить все регистры
00112A  4A 78 F6 44               tst.w    $f644.w              ; флаг смены палитры?
00112E  67 50                     beq.b    loc_001180           ; 0 → пропустить

; ═══ ПОДМЕНА ПАЛИТРЫ В СЕРЕДИНЕ КАДРА ═══
001130  41 F8 FA 80               lea.l    $fa80.w, a0    ; подводная палитра (64 цвета)
001134  23 FC C0 00 00 00 00 C0 00 04  move.l #$C0000000, $c00004.l  ; VDP: CRAM write, addr=0

; Запись 64 цветов (32 longword = 128 байт):
00113E  23 D8 00 C0 00 00         move.l   (a0)+, $c00000.l  ; цвет 0-1
001144  23 D8 00 C0 00 00         move.l   (a0)+, $c00000.l  ; цвет 2-3
; ... (30 раз)
001188  23 D8 00 C0 00 00         move.l   (a0)+, $c00000.l  ; цвет 62-63

; ═══ ОТКЛЮЧИТЬ ПОВТОРНЫЙ HBlank ═══
00118C  23 FC 00 00 8A DF 00 C0 00 04  move.l #$8ADF, $c00004.l
; VDP Register $0A = $DF (223) → HBlank counter = 223 строк
; = НЕ СРАБАТЫВАТЬ до конца кадра (one-shot)
```

### 26.2 Как работает эффект воды

**Принцип:** Экран Genesis состоит из 224 строк. VDP Register `$0A` задаёт, через сколько строк срабатывает HBlank-прерывание.

1. **VBlank (начало кадра):** Загружается обычная палитра из `$FB00` в CRAM. Register `$0A` = Y-координата линии воды (например, строка 128).
2. **HBlank (строка воды):** Прерывание срабатывает ровно на строке воды. Обработчик **заменяет всю палитру** (64 цвета) на подводную из `$FA80`.
3. **Блокировка:** Register `$0A` = `$DF` (223) → второго прерывания не будет до конца кадра.

**Результат:** Верхняя часть экрана (над водой) — обычные яркие цвета. Нижняя часть (под водой) — затемнённая/синяя палитра. **Без дополнительного оборудования**, чисто программный трюк.

### 26.3 Запись `$8B00` — режим скроллинга

```asm
; VDPSetupGame, $002E36
002E36  move.w   #$8B00, $c00004.l   ; VDP Reg $0B = $00
; Mode Register 3: horizontal scroll = FULL SCREEN
; Бит 1-0 = 00: один scroll offset для всего экрана
; (не per-line и не per-cell)
```

Sonic 1 использует **полноэкранный горизонтальный скроллинг** (не per-line). Параллакс фонов реализован через **отдельные scroll planes** (Plane A / Plane B), а не через per-line HScroll.

## 27. Путь обработки урона при 0 колец

### 27.1 Полная цепочка вызовов

```
TouchResponse ($01AE80)

    → Sonic_LoseRings_Sub ($01B098)  [rings == 0: DEATH]
      → Sonic_Dead ($013826)
        → Sonic_Restart ($01387E)
          → [subq.b #1, $fe12.w]    [lives -= 1]
```

### 27.2 Sonic_LoseRings — точка принятия решения ($01AFF4)

```asm
Sonic_LoseRings:
  01AFF4  4A 78 FE 2C               tst.w    $fe2c.w       ; ═══ ЕСТЬ ЩИТ? ═══
  01AFF8  66 22                     bne.b    loc_01B01C    ; ДА → потерять щит, НЕ кольца
  01AFFA  4A 78 FE 20               tst.w    $fe20.w       ; ═══ КОЛЬЦА > 0? ═══
  01AFFE  67 00 00 90               beq.w    loc_01B090    ; НОЛЬ → путь смерти!

  ; [rings > 0]: создать объект рассыпания колец ($37)
  01B002  jsr      FindFreeObj
  01B00A  move.b   #$37, $0(a1)     ; Object ID = Ring Scatter

loc_01B01C:
  ; Общий hurt path (и для щита, и для колец):
  01B01C  clr.w    $fe2c.w          ; убрать щит
  01B022  move.b   #$4, $24(a0)     ; routine = $04 (hurt state)
  01B032  move.w   #$fc00, $12(a0)  ; Y velocity = $FC00 (отскок вверх)
  01B038  move.w   #$fe00, $10(a0)  ; X velocity = $FE00 (отскок влево)
  01B066  move.b   #$1a, $1c(a0)    ; animation = $1A (hurt)
  01B06C  move.w   #$78, $30(a0)    ; invincibility timer = 120 кадров (2 сек)
```

### 27.3 Путь смерти при 0 колец ($01B090–$01B0EE)

```asm
loc_01B090:
  01B090  4A 78 FF FA               tst.w    $fffa.w       ; ═══ ФЛАГ БЕССМЕРТИЯ? ═══
  01B094  66 86                     bne.b    loc_01B01C    ; ≠ 0 → hurt (НЕ смерть!)

Sonic_LoseRings_Sub:
  01B098  4A 78 FE 08               tst.w    $fe08.w       ; ═══ INVINCIBILITY ACTIVE? ═══
  01B09C  66 4E                     bne.b    loc_01B0EC    ; ДА → полный пропуск (return)

  ; ═══ СМЕРТЬ ═══
  01B0A4  move.b   #$6, $24(a0)     ; routine = $06 (DEAD!)
  01B0B4  move.w   #$f900, $12(a0)  ; Y velocity = $F900 (сильный подброс)
  01B0BA  move.w   #$0, $10(a0)     ; X velocity = 0 (вертикально вверх)
  01B0CC  move.b   #$18, $1c(a0)    ; animation = $18 (death pose)
  01B0D2  bset.b   #$7, $2(a0)      ; high-priority sprite (поверх всех)
```

### 27.4 Дерево решений

```
УРОН ОТ ВРАГА/ШИПОВ
  ├── $FE2C ≠ 0 (ЩИТ)?
  │     └── ДА → потерять щит, animation=$1A, invincibility=120 кадров
  ├── $FE20 > 0 (КОЛЬЦА)?
  │     └── ДА → рассыпать кольца ($37), hurt recoil, sound $A3
  ├── $FFFA ≠ 0 (БЕССМЕРТИЕ)?
  │     └── ДА → hurt без смерти (вырезанная фича!)
  ├── $FE08 ≠ 0 (INVINCIBILITY)?
  │     └── ДА → полный пропуск (return $FF)
  └── ИНАЧЕ:
        └── СМЕРТЬ (routine=$06, animation=$18, Y vel=$F900)
```

**Ключевой момент:** Нет underflow-проверки на кольца! Если `$FE20 == 0`, это проверяется через `tst.w` (Z flag). Счётчик **не декрементируется** — он уже был 0.

## 28. Edge case: 1-Up при 0 жизней

### 28.1 Начисление жизни за кольца ($009C76)

```asm
; CollectRing — при сборе кольца:
009C76  52 78 FE 20               addq.w   #$1, $fe20.w   ; rings += 1

; Проверка на 100 колец:
009C84  0C 78 00 64 FE 20         cmpi.w   #$64, $fe20.w  ; rings == 100?
009C8C  08 F8 00 01 FE 1B         bset.b   #$1, $fe1b.w   ; set milestone bit 1
009C92  66 XX                     bne.b    skip           ; bit was 1 → УЖЕ ПОЛУЧЕНО
009C94  52 78 FE 12               addq.w   #$1, $fe12.w   ; ═══ LIVES += 1 ═══

; Проверка на 200 колец:
009C9C  0C 78 00 C8 FE 20         cmpi.w   #$c8, $fe20.w  ; rings == 200?
009CA4  08 F8 00 02 FE 1B         bset.b   #$2, $fe1b.w   ; set milestone bit 2
009CAC  66 XX                     bne.b    skip
009CB0  52 78 FE 12               addq.w   #$1, $fe12.w   ; LIVES += 1
```

### 28.2 Вычитание жизни при смерти ($0138A0)

```asm
; Sonic_Restart — при смерти:
0138A0  53 38 FE 12               subq.b   #$1, $fe12.w   ; ═══ LIVES -= 1 ═══
0138A4  66 2E                     bne.b    loc_0138D4     ; ≠ 0 → продолжить
; lives == 0:
0138AC  move.b   #$39, $d080.w    ; создать Game Over объект ($39)
0138B2  move.b   #$39, $d0c0.w    ; второй экземпляр
```

### 28.3 Анализ edge case

**Вопрос:** Что будет, если собрать 1-Up монитор при lives == 0?

**Ответ:** `addq.w #$1, $fe12.w` при 0 → lives = 1. **Стандартный инкремент, никаких проблем.**

**Настоящий edge case:** При `subq.b #$1, $fe12.w`:
- Операция `subq.b` работает с **байтом**, а `addq.w` — со **словом**
- Lives = 0: `subq.b #1, 0` → `$FF` (255) и Z flag = 0 (не ноль!)
- `bne.b loc_0138D4` **пройдёт** → Game Over **НЕ наступит**, lives = 255

**НО:** В практике это невозможно, т.к. смерть с 0 lives не может произойти — игрок должен **уже иметь** хотя бы 1 life чтобы играть. Начальное значение lives = 3 (`$FE12 = 3`).

**Реальная аномалия:** `subq.b` vs `addq.w` — **разные размеры операнда**. Это означает, что lives > 255 (если набрать через `addq.w`) используют **старший байт** `$FE12`, который `subq.b` **не трогает**. При lives = 256 (`$0100`), `subq.b #1` на младшем байте: `$00 - 1 = $FF`, итого `$01FF` (511). Lives **никогда не уменьшится** ниже 256 через `subq.b`!

## 30. Система палитровых переходов

### 30.1 PaletteFadeIn ($001DAC) — из чёрного

```asm
PaletteFadeIn:
  001DAC  move.w   #$3f, $f626.w    ; counter = 63 (64 цвета)

  ; Заполнить активную палитру чёрным:
  001DBA  lea.l    $fb00.w, a0      ; active palette
  001DC0  moveq    #$1f, d0         ; 32 longwords = 64 words = 64 цвета
  001DC2  moveq    #$0, d1
  001DC4  move.l   d1, (a0)+        ; write $00000000
  001DC6  dbra     d0, $1dc4        ; loop 32×

  ; Основной цикл (22 итерации):
  001DCA  move.w   #$15, d4         ; 21 шаг + 1 = 22 итерации ($15+1)

FadeIn_Loop:
  ; VBlank wait...
  001DDA  lea.l    $fb00.w, a0      ; текущая палитра
  001DDE  lea.l    $fb80.w, a1      ; целевая палитра
  001DE4  move.w   $f626.w, d0      ; количество цветов

FadeIn_Color:
  001DEA  move.w   (a0), d2         ; текущий цвет
  001DEC  move.w   (a1)+, d3        ; целевой цвет

  ; ═══ СРАВНЕНИЕ ПО КАНАЛАМ (BGR) ═══
  ; Синий канал (биты 1-3):
  001DEE  move.w   d3, d1
  001DF0  andi.w   #$e, d1          ; blue target
  001DF4  move.w   d2, d0
  001DF6  andi.w   #$e, d0          ; blue current
  001DFA  cmp.w    d1, d0
  001DFC  bcc.b    skip_blue        ; current >= target → пропуск
  001DFE  addi.w   #$2, (a0)+       ; ═══ BLUE += 2 ═══

  ; Зелёный канал (биты 5-7):
  001E06  andi.w   #$e0, d1         ; green target
  001E0A  andi.w   #$e0, d0         ; green current
  001E0E  cmp.w    d1, d0
  001E10  bcc.b    skip_green
  001E12  addi.w   #$20, (a0)+      ; ═══ GREEN += $20 ═══

  ; Красный канал (биты 9-11):
  001E1A  andi.w   #$e00, d1        ; red target
  001E1E  andi.w   #$e00, d0        ; red current
  001E22  cmp.w    d1, d0
  001E24  bcc.b    skip_red
  001E26  addi.w   #$200, (a0)+     ; ═══ RED += $200 ═══
```

**Алгоритм:** Каждый из 64 цветов обрабатывается **поканально**. На каждом из 22 шагов каждый канал (B/G/R) увеличивается на 1 уровень (`+$2`, `+$20`, `+$200`). Genesis поддерживает 3 бита на канал (8 уровней: 0–$E), поэтому 7 шагов достаточно для полного fade одного канала. 22 шага = запас на случай, когда все три канала нужно довести от 0 до максимума ($EEE).

### 30.2 PaletteFadeOut ($001E52) — в чёрный

```asm
PaletteFadeOut:
  001E52  move.w   #$3f, $f626.w    ; 64 цвета
  001E58  move.w   #$15, d4         ; 22 шага

FadeOut_Color:
  ; Обратная логика: вычитаем от текущих цветов к нулю
  ; Blue: subi.w #$2, (a0)+
  ; Green: subi.w #$20, (a0)+
  ; Red: subi.w #$200, (a0)+
```

Зеркальный алгоритм FadeIn: вычитает по канальным компонентам до нуля.

### 30.3 PaletteWhiteIn ($001ED0) — из белого

```asm
PaletteWhiteIn:
  001ED0  move.w   #$3f, $f626.w
  ; Заполнить палитру БЕЛЫМ:
  001EDE  move.w   #$eee, d1        ; ═══ $0EEE = БЕЛЫЙ ═══
  001EE2  move.w   d1, (a0)+        ; 64 цвета × $0EEE
  ; ...
  ; Далее цикл ВЫЧИТАНИЯ от белого к целевой палитре
  ; (обратная логика FadeIn)
```

Используется при **выходе из Special Stage** — экран вспыхивает белым, затем плавно проявляется палитра уровня. Цвет `$0EEE`: R=7, G=7, B=7 — максимальная яркость всех каналов.

### 30.4 PaletteWhiteOut ($001F7C) — в белый

Зеркальный алгоритм WhiteIn: каждый канал **увеличивается** до `$E`, создавая эффект вспышки → белый экран.

### 30.5 Специальная обработка Labyrinth Zone

```asm
; Внутри PaletteFadeIn, после основного цикла:
001E00  cmpi.w   #$1, $fe10.w      ; Zone == 1 (Labyrinth)?
001E0A  lea.l    $fa80.w, a0       ; → fade также подводную палитру
001E0E  lea.l    $fa00.w, a1       ; → из целевой подводной
```

В LZ **две палитры** фейдятся одновременно: обычная (`$FB00→$FB80`) и подводная (`$FA80→$FA00`). Это обеспечивает корректный плавный переход при загрузке уровня с водой.

## 31. Генератор псевдослучайных чисел (PRNG)

### 31.1 Алгоритм ($0029B4)

```asm
RandomNumber:
  0029B4  30 38 F6 36               move.w   $f636.w, d0    ; seed (low word)
  0029B8  66 06                     bne.b    loc_0029C0     ; ≠ 0?
  0029BA  30 3C 2A 6D               move.w   #$2a6d, d0     ; default = $2A6D
  0029C0  32 00                     move.w   d0, d1         ; d1 = seed
  0029C2  E5 41                     asl.w    #$2, d1        ; d1 = seed × 4
  0029C4  D2 40                     add.w    d0, d1         ; d1 = seed × 5
  0029C6  E7 41                     asl.w    #$3, d1        ; d1 = seed × 40
  0029C8  D2 40                     add.w    d0, d1         ; d1 = seed × 41
  0029CA  30 01                     move.w   d1, d0
  0029CC  48 41                     swap     d1             ; d1 = swap(d1)
  0029CE  D0 41                     add.w    d1, d0         ; d0 = low(d1) + high(d1)
  0029D0  31 C0 F6 36               move.w   d0, $f636.w   ; новый seed
  0029D4  4E 75                     rts
```

### 31.2 Математический анализ

```
seed_new = ((seed * 5) * 8 + seed)  →  seed * 41
result = low_word(seed*41) + high_word(seed*41)
PRNG_state = swap_words(result, upper_half)
```

**Множитель = 41.** Простая LCG (Linear Congruential Generator) без модуля (переполнение word = mod 65536 implicit).

**Начальное значение:** `$2A6D` = 10861. Если seed = 0, используется `$2A6D`. Seed хранится в `$F636.w`.

### 31.3 Вызывающие функции

| Функция | Адрес | Применение |
|---|---|---|
| `Sonic_WallCollide` | $013XXX | Случайная частица при столкновении |
| `Boss_MakeExplosion` | $01XXXX | Позиция осколков босса |
| `Obj39_GameOver_Main` | $01XXXX | Анимация Game Over |
| `Obj37_RingScatter` | $01ACFA | Направление рассыпанных колец |
| `Obj_RingScatter_Move` | $01AC84 | Случайная позиция частиц капсулы |

### 31.4 Качество PRNG

Множитель 41 даёт **короткий период** (≤65536 для 16-bit seed). Для игровых целей достаточно: рассыпание колец, взрывы боссов и Game Over анимации не требуют криптографической случайности. Однако последовательность **полностью детерминирована** — при одинаковом seed получается одинаковый разброс колец. TAS-ы (Tool-Assisted Speedruns) используют это для манипуляции RNG.

## 32. Диспетчер объектов (ExecuteObjects)

### 32.1 Главный цикл ($00D338)

```asm
ExecuteObjects:
  00D338  lea.l    $d000.w, a0      ; начало массива объектов
  00D33C  moveq    #$7f, d7         ; 128 слотов (0–$7F)
  00D33E  moveq    #$0, d0

ExecObj_Loop:
  00D340  move.b   (a0), d0         ; Object ID = byte 0
  00D342  67 12                     beq.b    ExecObj_Next  ; ID == 0 → пустой слот
  00D344  D0 40                     add.w    d0, d0        ; d0 × 2
  00D346  D0 40                     add.w    d0, d0        ; d0 × 4
  00D348  movea.l  $d37e(pc, d0.w), a1  ; таблица указателей
  00D34C  jsr      (a1)             ; ═══ ВЫЗОВ ФУНКЦИИ ОБЪЕКТА ═══

ExecObj_Next:
  00D350  lea.l    $40(a0), a0      ; следующий слот (+64 байта)
  00D354  dbra     d7, ExecObj_Loop ; повторить 128 раз
```

### 32.2 Структура объектных слотов

```
$D000–$D03F  : Слот 0  — Sonic ($01)
$D040–$D07F  : Слот 1  — HUD ($21)
$D080–$D0BF  : Слот 2  — зарезервирован (Game Over $39 и т.д.)
$D0C0–$D0FF  : Слот 3  — зарезервирован
...
$D7C0–$D7FF  : Слот 31 — последний зарезервированный
$D800–$D83F  : Слот 32 — первый ДИНАМИЧЕСКИЙ
...
$EFC0–$EFFF  : Слот 127 — последний динамический
```

- **128 слотов** × 64 байта = **8192 байт** ($D000–$EFFF)
- Слоты 0–31: **зарезервированные** (системные объекты)
- Слоты 32–127: **96 динамических** (FindFreeObj ищет здесь)

### 32.3 Оптимизация при смерти Sonic

```asm
; Внутри ExecuteObjects — проверка смерти:
00D356  cmpi.b   #$6, $d024.w      ; Sonic routine >= 6 (dead)?
00D35C  bcs.b    ExecObj_AllObjects ; < 6 → обработать все 128
; Sonic мёртв:
; Обработать только первые 32 слота ($D000–$D7FF)
; Остальные 96 ($D800–$EFFF) → только DisplaySprite (рендер без логики)
```

Когда Sonic мёртв, динамические объекты **замораживаются** — выполняется только их отрисовка. Это объясняет, почему при смерти Sonic враги и платформы «застывают».

### 32.4 Таблица указателей объектов ($00D37E)

Таблица содержит 4-байтовые longword адреса для каждого Object ID. При ID = `$01` (Sonic): offset = `$01 × 4 = $04`, таблица[$04] = адрес `Sonic_Main`.

**Отсутствие bound check:** Если Object ID > максимального, `d0 × 4` выходит за пределы таблицы и читает **произвольные данные** как адрес функции → crash или непредсказуемое поведение.

### 32.5 Obj_JumpTable — подобъектный роутинг ($015730)

```asm
Obj_JumpTable:
  015730  moveq    #$0, d0
  015732  move.b   $24(a0), d0      ; sub-routine из поля $24
  015736  move.w   $6(a1, d0.w), d0 ; offset из таблицы
  01573A  jmp      $6(a1, d0.w)     ; jump
```

`Obj_JumpTable` — утилита для маршрутизации подсостояний объекта. Каждый объект хранит текущий sub-routine в `$24(a0)`, увеличивая на 2 при переходе к следующему состоянию (`addq.b #2, $24(a0)`).

## 33. Система анимации Sonic

### 33.1 Архитектура ($0139C4)

```asm
Sonic_Animate:
  0139C4  lea.l    $13b48.l, a1     ; ═══ ТАБЛИЦА АНИМАЦИЙ ═══
  0139CA  moveq    #$0, d0
  0139CC  move.b   $1c(a0), d0      ; текущая анимация ID
  0139D0  cmp.b    $1d(a0), d0      ; изменилась?
  0139D4  beq.b    loc_0139E6       ; нет → продолжить текущий кадр

  ; Анимация изменилась → сброс:
  0139D6  move.b   d0, $1d(a0)      ; сохранить новый ID
  0139DA  move.b   #$0, $1b(a0)     ; frame index = 0
  0139E0  move.b   #$0, $1e(a0)     ; frame timer = 0

loc_0139E6:
  ; Индексация таблицы:
  0139E6  add.w    d0, d0            ; ID × 2
  0139E8  adda.w   (a1, d0.w), a1   ; a1 += offset → указатель на данные анимации
  0139EC  move.b   (a1), d0         ; байт 0 = frame duration / special flag
```

### 33.2 Формат данных анимации

Каждая запись в таблице `$13B48`:
- **Байт 0:** Duration / Flag
  - `$00–$7F` — длительность кадра в фреймах
  - `$FF` — loop (вернуться к байту 1)
  - `$FE` — loop back: следующий байт = сколько кадров назад
  - `$FD` — change animation: следующий байт = новый animation ID
  - `$80–$FE` (bit 7 set) — **скоростная анимация** (walk/run/roll)
- **Байты 1+:** Sprite frame indices (terminated by $FF/$FE/$FD)

### 33.3 Скоростная анимация Walk/Run ($013A52–$013AD0)

```asm
; Скоростная анимация (bit 7 of duration byte set):
loc_013A52:
  ; Вычисление направления из угла наклона:
  013A60  move.b   $26(a0), d0      ; floor angle
  013A68  andi.b   #$1, d2          ; facing direction
  013A6E  not.b    d0               ; инвертировать если facing left
  013A70  addi.b   #$10, d0         ; offset
  013A8E  lsr.b    #$4, d0          ; d0 = angle / 16 → 0–15
  013A90  andi.b   #$6, d0          ; d0 &= 6 → 0,2,4,6

  ; Выбор набора кадров на основе скорости:
  013A94  move.w   $14(a0), d2      ; ground speed
  013A98  bpl.b    skip             ; abs value
  013A9A  neg.w    d2

  013A9C  lea.l    $13b8e.l, a1     ; ═══ RUN FRAMES ═══
  013AA2  cmpi.w   #$600, d2        ; speed >= $600?
  013AA6  bcc.b    use_run          ; ДА → run animation

  013AA8  lea.l    $13b86.l, a1     ; ═══ WALK FRAMES ═══
  ; walk: d0 = angle*1.5 (3 кадра на направление)
  013AAE  move.b   d0, d1
  013AB0  lsr.b    #$1, d1          ; d1 = d0/2
  013AB2  add.b    d1, d0           ; d0 = d0*1.5
```

### 33.4 Формула длительности кадра

```asm
  ; Frame duration = f(speed):
  013AB8  neg.w    d2               ; d2 = -speed
  013ABA  addi.w   #$800, d2        ; d2 = $800 - speed
  013ABE  bpl.b    skip
  013AC0  moveq    #$0, d2          ; if negative → d2 = 0

  013AC2  lsr.w    #$8, d2          ; d2 = ($800 - speed) / 256
  013AC4  move.b   d2, $1e(a0)      ; ═══ FRAME DURATION ═══
```

**Формула:** `duration = max(0, ($800 - |speed|)) >> 8`

| Ground Speed | Duration (кадры) | FPS анимации |
|---|---|---|
| $000 (стоя) | 8 | ~7.5 |
| $200 (медленно) | 6 | ~10 |
| $400 (средне) | 4 | ~15 |
| $600 (быстро) | 2 | ~30 |
| $800+ (макс) | 0 | **60** (каждый кадр) |

### 34.4 Obj11_Depress — динамический прогиб ($01057A)

```asm
Obj11_Depress:
  ; Расширенная физика моста:
  01057A  move.b   $38(a0), d0      ; active flag
  ; $38(a0) = активирован ли прогиб
  ; $3A(a0) = текущий уровень прогиба (0–$80)
  ; $3C(a0) = trigger index

  ; Инкремент/декремент прогиба:
  ; Когда Sonic стоит на мосту: $3A += 2 каждый кадр
  ; Когда Sonic сходит: $3A -= 2 каждый кадр
  ; Максимум: $3A = $80 (128)
```

**Система:** Прогиб нарастает **постепенно** (по 2 единицы за кадр), а не мгновенно. Когда Sonic прыгает или сходит с моста, мост **плавно возвращается** к исходной форме. Разные точки входа (`Obj11_Depress + offset`) обрабатывают мосты разной ширины ($10, $30, $50, $70 пикселей).

**Формула итогового Y смещения:**
```
Y_offset = Bridge_HeightData[Sonic_X_relative / 2] * depression_level / 128
```

Где `depression_level` — анимированное значение 0→$80 при наступании, $80→0 при освобождении.

## 36. Механизм входа в Special Stage

### 36.1 Giant Ring — проверки ($0166E0)

```asm
; Giant Ring Object:
0166E0  ; Проверка расстояния до Sonic:
  move.w   $d008.w, d0      ; Sonic X
  sub.w    $8(a0), d0       ; relative X
  cmpi.w   #$10, d0         ; |ΔX| < $10 (16 пикселей)?

  move.w   $d00c.w, d0      ; Sonic Y
  sub.w    $c(a0), d0       ; relative Y
  subi.w   #$20, d0         ; offset $20
  cmpi.w   #$40, d0         ; |ΔY| < $40 (64 пикселя)?

; Проверка флага "уже вошёл":
  tst.w    $f7c8.w          ; already entered?
  bne.b    skip             ; ДА → пропуск

; ═══ ПРОВЕРКА 50 КОЛЕЦ ═══
016700  cmpi.w   #$32, $fe20.w    ; rings >= 50 ($32)?
016704  bcs.b    skip             ; < 50 → НЕ ВХОДИТЬ

  move.b   #$81, $f7c8.w    ; set entered flag
  move.b   #$2, $1c(a0)     ; Sonic animation = look up
  move.w   #$800, speed     ; Sonic launch speed
  clr.w    $10(a0)          ; X velocity = 0
  clr.w    $12(a0)          ; Y velocity = 0
```

### 36.2 Условия входа

1. **Positional:** Sonic в пределах 16×64 пикселей от Giant Ring
2. **Not yet entered:** `$F7C8 == 0`
3. **50+ rings:** `$FE20 >= $32` — **только для subtype $07**!
4. **After entry:** Sonic получает animation «look up», запускается вверх со скоростью `$800`

### 36.3 SS_Results — 50 колец в Special Stage ($01BD52)

## 37. Коммуникация с Z80 Sound Driver

### 37.1 SoundDriverLoad ($001352)

```asm
SoundDriverLoad:
  001352  move.w   #$100, $a11100.l   ; ═══ Z80 BUS REQUEST ═══
  ; Ждать готовности Z80:
  001360  btst.b   #$0, $a11100.l     ; bus granted?
  001366  beq.b    $1360              ; нет → retry

  ; Z80 RESET:
  001368  move.w   #$100, $a11200.l   ; assert reset
  ; Загрузка звукового драйвера:
  001376  lea.l    $72e7c.l, a0       ; ROM: сжатые данные драйвера
  00137C  lea.l    $a00000.l, a1      ; Z80 RAM: $A00000
  001382  jsr      KosDec             ; Kosinski decompression

  ; Запуск Z80:
  001388  move.w   #$0, $a11200.l     ; release reset
  00138E  move.w   #$0, $a11100.l     ; ═══ Z80 BUS RELEASE ═══
```

**Протокол:**
1. Запросить Z80 bus (`$A11100 = $100`)
2. Подождать grant (бит 0 `$A11100`)
3. Assert Z80 reset (`$A11200 = $100`)
4. Скопировать звуковой драйвер из ROM `$72E7C` в Z80 RAM через Kosinski decompression
5. Release reset → Z80 начинает выполнение драйвера
6. Release bus

### 37.2 PlaySound / PlaySound_Special

```asm
; PlaySound ($001396) — обычные звуки:
001396  move.b   d0, $f601.w     ; записать ID в очередь

; PlaySound_Special ($00139C) — специальные:
00139C  move.b   d0, $f609.w     ; записать в спец. очередь
```

Звуки передаются через **RAM mailbox**: 68K записывает ID в `$F601`/`$F609`, а SoundDriver_Main на стороне 68K считывает и обрабатывает при следующем VBlank.

### 37.3 SoundDriver_Main ($071B4C)

```asm
SoundDriver_Main:
  ; ═══ 68K-СТОРОНА ЗВУКОВОГО ДРАЙВЕРА ═══
  071B4C  move.w   #$100, $a11100.l   ; Z80 BUS REQUEST
  ; Wait loop:
  071B5A  btst.b   #$7, $a01ffd.l     ; Z80 ready flag?
  071B6C  beq.b    loc_071B82         ; ready → process
  ; Не готов:
  071B6E  move.w   #$0, $a11100.l     ; release bus
  071B76  nop                          ; yield time
  071B78  nop
  071B7A  nop
  071B7C  move.w   #$100, $a11100.l   ; re-request
  071B80  bra.b    SoundDriver_Main   ; ═══ RETRY (бесконечный цикл!) ═══

loc_071B82:
  ; Z80 ready — обработка звука:
  071B82  jsr      SndDrv_TempoUpdate  ; обновить темп
  071B88  jsr      SndDrv_PlayMusic    ; воспроизвести музыку
  071B8E  jsr      SndDrv_UpdateSFX    ; обновить SFX
  071B94  jsr      SndDrv_UpdateFM     ; обновить FM каналы
  071B9A  move.w   #$0, $a11100.l     ; Z80 BUS RELEASE
```

**Критический момент:** Бит 7 `$A01FFD` — флаг готовности Z80. Если Z80 зависнет и никогда не установит этот бит, 68K войдёт в **бесконечный polling loop**: запрос → проверка → release → nop → re-request → retry. **Нет timeout**, игра зависнет навсегда.

### 37.4 Архитектура каналов

```
6 FM каналов  (YM2612): каждый $30 байт
3 PSG канала  (SN76489): каждый $30 байт
1 DAC канал   (PCM через YM2612 Ch6)
─────────────────────────────────
10 каналов × $30 = $1E0 байт RAM
```

**Интересные находки:**

### 38.4 Sonic_DoLevelBound — ограничение позиции ($013498)

```asm
Sonic_DoLevelBound:

  013498  move.w   $f728.w, d0    ; left X limit
  01349E  addi.w   #$10, d0       ; +16 (offset для центра спрайта)
```

**Vertical velocity cap:** Y скорость ограничивается **только при выходе за границу** уровня, не при обычном падении (см. баг 2.13 — `ObjectFall` не имеет Y velocity cap).

```c
/* 01AF76 */      $f7d0.w += #$2;          // += 2 за каждого врага

/* 01AF86 */      // lookup table: 10, 20, 50, 100
/* 01AF8A */      compare($f7d0.w, #$20);   // цепочка >= 16 врагов?
/* 01AF92 */      d0 = #$3e8;              // ═══ 10000 ОЧКОВ! ═══
```

**Скрытая механика:** `$F7D0` — счётчик цепочки уничтожений (инкремент +2). Таблица очков:

| Враги подряд | Очки |
|---|---|
| 1 | 100 |
| 2 | 200 |
| 3 | 500 |
| 4+ | 1,000 |
| **16+** | **10,000** ← скрытый бонус! |

Счётчик **сбрасывается в 0** при касании земли. Для получения 10,000 за каждого врага нужно уничтожить **16 врагов не касаясь земли** — практически невозможно без debug mode.

### 4.3 Обход скоростного лимита (Speed Cap Bypass)

| | |
|---|---|
| **Адрес ROM** | `$012D3A` (cap), `$01340C` (spring) |
| **Строка** | 18078, 19795 |

**Нормальный лимит:**
```c
/* 012D3A */      $f760.w = #$600;         // макс скорость = $600
/* 012D40 */      $f762.w = #$c;           // ускорение = $0C
/* 012D46 */      $f764.w = #$80;          // торможение = $80
```

**Под водой (LZ):**
```c
/* 012DA4 */      $f760.w = #$300;         // ½ скорости
/* 012DAA */      $f762.w = #$6;           // ½ ускорения
/* 012DB0 */      $f764.w = #$40;          // ½ торможения
```

**Пружина / отскок от стены:**
```c
/* 01340C */      d2 = #$680;              // ═══ $680 > $600! ═══
```

**3 способа обхода Speed Cap:**

1. **Пружины** — устанавливают скорость напрямую в `$10(a0)` / `$12(a0)`, минуя переменную `$F760`. Значение `$680` превышает cap на `$80`.
2. **Склоны** — при Rolling по склону гравитационная составляющая добавляется **до** проверки лимита, позволяя накопить скорость > `$600`.
3. **Выход из воды** — скорость under-водой ×2 при всплытии через `ASR` восстановление, если Sonic вышел из воды на большой скорости.

> Cap применяется **только** в `Sonic_Move` при ускорении. Абсолютное значение `$10(a0)` не ограничивается.

### 4.11 Диспетчер объектов (Jump Table)

| | |
|---|---|
| **Адрес ROM** | `$00D338` (ExecuteObjects), `$00D37E` (таблица) |
| **Строка ASM** | 15310 |

```asm
ExecuteObjects:
  00D338  lea.l      $d000.w, a0       ; первый объект
  00D33C  moveq      #$7f, d7          ; 128 слотов
loc_00D348:
  00D348  move.b     (a0), d0          ; ID объекта (1 байт)
  00D34C  add.w      d0, d0            ; × 2
  00D34E  add.w      d0, d0            ; × 4

  00D354  jsr        (a1)              ; ВЫЗОВ!
```

- 128 слотов × 64 байта = **8 КБ** объектной RAM (`$D000–$EFFF`)

### 5.1 ROM Header и проверка Checksum

| | |
|---|---|
| **Адрес ROM** | `$000100` (header), `$00038E` (checksum) |

**ROM Header** (`$100–$1FF`):
- `$100`: "SEGA MEGA DRIVE " (16 байт)
- `$110`: "(C)SEGA 1991.APR" — дата: **апрель 1991**
- `$120`: Название (JP): "SONIC THE HEDGEHOG"
- `$150`: Название (US/EU): "SONIC THE HEDGEHOG"
- `$180`: Серийный номер: "GM 00001009-00"
- `$18E`: **Checksum** (16-бит)
- `$1A4`: ROM range: `$000000–$07FFFF` (512 КБ)

**Проверка Checksum:**
```c
/* 00038E */      a0 = &$200.l;              // начало проверяемых данных
/* 000394 */      d0 = &$7ffff.l;            // конец ROM
/* 00039A */      d0 -= a0;                  // размер для проверки
/* 00039E */      d0 >>= #$1;               // / 2 (word-суммирование)
  loc_0003A4:
/* 0003A4 */      d1 += (a0)+;              // суммировать все words
/* 0003A6 */      if (!ra && --d0 != -1) goto loc_0003A4;
/* 0003AA */      compare(d1, $18e.l);       // сравнить с хедером
/* 0003B0 */      if (Z == 1) goto loc_0003D4;  // совпало → OK
/* 0003B2 */      $fc44.w = #$a;            // код ошибки = $A
```

**При несовпадении:**
```c
  loc_0003D4:
/* 0003D4 */      $c00000.l = #$e;          // CRAM = $000E (КРАСНЫЙ ЭКРАН)
/* 0003D8 */      goto loc_0003D4;           // ═══ БЕСКОНЕЧНЫЙ ЦИКЛ ═══
```

> При модификации ROM (ромхаки) без обновления checksum → **красный экран смерти** с бесконечным зацикливанием.

### 5.2 HBlank трюк с палитрой воды

| | |
|---|---|
| **Адрес ROM** | `$001126` (HBlank handler) |

В Labyrinth Zone линия воды реализована через **прерывание горизонтальной развёртки (HBlank)**:

1. VBlank устанавливает VDP регистр `$8Axx` = строка, где начинается вода
2. Когда VDP достигает этой строки → **HBlank прерывание** на `$001126`
3. Handler **подменяет всю палитру CRAM** (64 цвета) за один HBlank
4. Устанавливает `$8ADF` — **one-shot** (больше не прерывать до следующего VBlank)

**Результат:** Верхняя часть экрана рисуется обычной палитрой, нижняя (под водой) — палитрой с синим оттенком. Всё за **один кадр**, без двойной буферизации.

### 5.4 Операндный баг Lives (byte vs word)

| | |
|---|---|
| **Адрес ROM** | `$009CA4` (добавление), `$0138A0` (вычитание) |

```asm
; Добавить жизнь (word):
009CA4  addq.w  #1, $fe12.w      ; lives += 1 (WORD операция, 16 бит)

; Убрать жизнь (byte):
0138A0  subq.b  #1, $fe12.w      ; lives -= 1 (BYTE операция, 8 бит!)
```

**Баг:** Добавление = `addq.w` (16 бит), вычитание = `subq.b` (8 бит). Если lives > 255 (`$FF`):
- `addq.w` корректно увеличивает word
- `subq.b` уменьшает только **младший байт**, не затрагивая старший

При lives = `$0100` (256): `subq.b` → `$01FF` (511) вместо `$00FF` (255). Жизни **увеличиваются** вместо уменьшения!

### 5.6 Система палитрового фейда

| | |
|---|---|
| **Адрес ROM** | `PaletteFadeIn` / `PaletteFadeOut` |

**Алгоритм:**
- Палитра = 4 линии × 16 цветов = **64 цвета**
- Формат цвета: `$0BGR` (4 бита на канал, 12 бит на цвет)
- Fade = **7 шагов** × 3 канала (B, G, R) = **21 операция** на цвет

### 5.7 ГПСЧ (Random Number Generator)

| | |
|---|---|
| **Адрес ROM** | `RandomNumber` |

```
Алгоритм: Linear Congruential Generator
  seed = seed × 41 + $2A6D
  return d0 = seed (low word = "random" число)
```

- **Множитель:** 41 (`$29`)
- **Инкремент:** `$2A6D` (10861)
- **Период:** ≤ 65,536 (16-бит seed)
- **Полностью детерминирован** — при одинаковом seed → одинаковая последовательность

> **TAS-эксплойт:** Зная seed, можно предсказать ВСЕ "случайные" события (позиции врагов, поведение боссов, итемы из мониторов на random-уровнях).

### 5.8 Анимационная система Sonic

**Порог Walk → Run:**
```c
compare($10(a0), $f760.w);    // скорость == speed cap?
// speed cap = $600 → Run animation
// speed cap < $600 → Walk animation
```

**Run-анимация видна ТОЛЬКО если:**
1. Speed Shoes активны (cap = `$A00`)
2. На склоне при rolling (скорость > `$600` по баг-обходу cap)

**Длительность кадра ходьбы:**
```
frame_duration = ($800 - |speed|) >> 8
```
При скорости `$600` → `($800 - $600) >> 8 = $200 >> 8 = 2` кадра на фрейм → максимально быстрая анимация.

### 5.12 Z80 Sound Driver — Deadlock

| | |
|---|---|
| **Адрес ROM** | `$001294` (Z80 bus request) |

```c
/* 001294 */      $a11100.l = #$100;       // запросить Z80 bus
  loc_00129C:
/* 00129C */      test_bit($a11100.l, #$0); // Z80 освободил bus?
/* 0012A4 */      if (Z == 1) goto loc_00129C; // НЕТ → ЖДАТЬ БЕСКОНЕЧНО
```

**Баг:** Нет timeout'а на ожидание Z80 bus. Если Z80 зависает (аппаратный сбой, corruption RAM) → 68000 входит в **бесконечный цикл опроса** → **полный deadlock** обоих процессоров.

### 7.1 Исправления неточностей в наших находках

#### ⚠️ 7.1.1 Level Select — уточнение последовательности ввода

**Мы написали:** "Up, Down, Left, Right на экране SEGA"

**Правильно:** Последовательность **Up, Down, Left, Right** вводится на **титульном экране** (Title Screen), не на SEGA screen. На экране SEGA нужно быстро нажать **C, C, C, C, C, C** (6 раз) — это устанавливает скрытый флаг в RAM. Затем на титульном экране — Up, Down, Left, Right, после чего зажать **A** и нажать **Start**.

В **японской версии (Rev 0):** последовательность другая — Up, C, Down, C, Left, C, Right, C на Title Screen, затем Start при зажатом A.

#### ⚠️ 7.1.2 Debug Mode — точная последовательность Sound Test

**Мы написали:** "Выбрать Sound Test + комбо кнопок (C + Start)"

**Правильно (US/EU v1.0):** В Level Select Sound Test нужно воспроизвести звуки в определённом порядке: **$01, $09, $09, $01, $00, $06, $02, $03**. После каждого нажимать C для воспроизведения. Затем нажать A+Start для входа в уровень с включённым Debug Mode. Наш код правильно показывает проверку `$FE08`, но мы не идентифицировали **конкретную последовательность** байтов из ROM.

#### ⚠️ 7.1.3 Spike Bug — уточнение механизма

**Мы написали:** "NOP на $01AFEA — место, где должна быть проверка"

**Правильно (по s1disasm):** Spike Bug возникает не из-за NOP, а потому что шипы проверяют **invincibility timer** `$30(a0)`, но этот таймер **сбрасывается в 0** когда Sonic касается пола. Если Sonic отскакивает от шипов и **касается пола** до повторного контакта с шипами — таймер уже 0, и шипы снова наносят урон.

Точный механизм из s1disasm:
```asm
; В Sonic_ResetOnFloor:
    clr.b   $30(a0)    ; ← СБРОС INVINCIBILITY TIMER при касании пола!
```
Враги этому не подвержены, потому что у них **другой код обработки столкновений** (не проверяет `$30`).

#### ⚠️ 7.1.4 Вырезанный флаг бессмертия ($FFFA) — уточнение

**Мы написали:** "Нигде не устанавливается"

**Уточнение:** По данным TCRF и s1disasm, `$FFFA` — это **debug cheat flag**, который устанавливается через специальную комбинацию на экране Level Select в **японской версии**. В US/EU версии код установки **вырезан**, но проверка (`tst.b $FFFA`) осталась. Наша идентификация корректна для US/EU ROM, но стоило указать его оригинальное назначение.

#### ⚠️ 7.1.5 Монитор ID 0 (Eggman Monitor)

**Мы написали:** "ID 6 = 'S' монитор, неактивен"

**Правильно:** Монитор типа **$00** (Eggman/Robotnik monitor) при ударе **наносит урон Sonic** вместо бонуса. Это задокументировано в s1disasm как `Obj26_Eggman`. Монитор типа $06 — **Static/Sonic icon** (пустой эффект).

### 8.2 ⚠️ КРИТИЧЕСКОЕ ИСПРАВЛЕНИЕ: Механизм Debug Mode в Rev00

**Ранее описанное (неверно для этого ROM):**
> "В Sound Test введите 01, 09, 09, 01, 00, 06, 02, 03"

**Фактический механизм (подтверждён ROM-анализом):**

Последовательность Sound Test **отсутствует** в ROM. Поиск байтовых паттернов `01 09 09 01 00 06 02 03` и word-версии `00 01 00 09 00 09 00 01 00 00 00 06 00 02 00 03` — **0 совпадений**.

#### Реальная активация Debug Mode в Rev00:

**Шаг 1: Чит-последовательность на титульном экране ($31B6–$3210)**

```m68k
$31B6  tst.b      $fff8.w           ; Проверка региона
$31BA  bpl.b      $31c4             ; Не Японская → US/EU
$31BC  lea.l      $33aa.l, a0       ; Японская последовательность
$31C2  bra.b      $31ca
$31C4  lea.l      $33a4.l, a0       ; US/EU последовательность
$31CA  move.w     $ffe4.w, d0       ; Текущая позиция в последовательности
$31CE  adda.w     d0, a0
$31D0  move.b     $f605.w, d0       ; Ново-нажатые кнопки
$31D4  andi.b     #$f, d0           ; Маска D-pad (Up/Down/Left/Right)
$31D8  cmp.b      (a0), d0          ; Сравнение с ожидаемым
$31DA  bne.b      $3210             ; Не совпало → сброс
$31DC  addq.w     #$1, $ffe4.w      ; Следующая позиция
```

**Последовательность в ROM ($33A4):**

| Байт | Значение | Кнопка D-pad |
|---|---|---|
| `$01` | бит 0 | **Up** |
| `$02` | бит 1 | **Down** |
| `$04` | бит 2 | **Left** |
| `$08` | бит 3 | **Right** |
| `$00` | конец | *End marker* |
| `$FF` | терминатор | — |

Японская последовательность ($33AA) **идентична** US/EU.

**Шаг 2: Система вариантов по C-кнопке ($31E4–$3206)**

При завершении последовательности проверяется счётчик нажатий C (`$FFE6`):

```m68k
$31E4  lea.l      $ffe0.w, a0       ; Базовый адрес флагов читов
$31E8  move.w     $ffe6.w, d1       ; Счётчик C-кнопки
$31EC  lsr.w      #$1, d1           ; C_count / 2
$31EE  andi.w     #$3, d1           ; (C_count / 2) & 3 → вариант 0–3
$3200  move.b     #$1, (a0, d1.w)   ; $FFE0 + variant = 1
```

| Нажатий C | Формула | Вариант | Устанавливает | Эффект |
|---|---|---|---|---|
| 0–1 | 0>>1 & 3 = 0 | 0 | `$FFE0 = 1` | **Level Select** |
| 2–3 | 2>>1 & 3 = 1 | 1 | `$FFE1 = 1` | **Сброс в титульный при A на паузе** |
| 4–5 | 4>>1 & 3 = 2 | 2 | `$FFE2 = 1` | **Debug Mode доступен** |
| 6–7 | 6>>1 & 3 = 3 | 3 | `$FFE3 = 1` | Зарезервировано (нет эффекта) |

> **Кумулятивный механизм:** Счётчик `$FFE6` НЕ сбрасывается между вводами. После завершения последовательности позиция `$FFE4` сбрасывается на 0 (т.к. `$FFE4 ≠ 9`, проверка в `$3214`), позволяя вводить чит повторно. Каждый ввод устанавливает **следующий** вариант.

**Шаг 3: Вход в Level Select ($3242–$3256)**

```m68k
$3238  andi.b     #$80, $f605.w     ; Start нажат?
$323E  beq.w      $317c             ; Нет → цикл
$3242  tst.b      $ffe0.w           ; Level Select включён?
$3246  beq.w      $3342             ; Нет → обычный старт
$324A  btst.b     #$6, $f604.w      ; A зажата?
$3250  beq.w      $3342             ; Нет → обычный старт
$3254  moveq      #$2, d0           ; Game Mode = Level Select
```

**Шаг 4: Level Select устанавливает $FFFA ($3982–$3990)**

```m68k
$3982  tst.b      $ffe2.w           ; Debug флаг активен? (вариант 2)
$3986  beq.b      $3996             ; Нет → пропуск
$3988  btst.b     #$6, $f604.w      ; A зажата при входе?
$398E  beq.b      $3996             ; Нет → пропуск
$3990  move.b     #$1, $fffa.w      ; ★ Установить Debug Available!
```

**Шаг 5: In-game активация ($012C4C)**

```m68k
$12C3E  tst.w      $fffa.w          ; Debug доступен?
$12C42  beq.b      $12c58           ; Нет → пропуск
$12C44  btst.b     #$4, $f605.w     ; C нажата?
$12C4A  beq.b      $12c58           ; Нет → пропуск
$12C4C  move.w     #$1, $fe08.w     ; ★ ВКЛЮЧИТЬ Debug Mode!
```

#### Полная инструкция для Rev00:

1. На титульном экране: **Up, Down, Left, Right** → `$FFE0 = 1` (Level Select)
2. Нажать **C** дважды, снова **Up, Down, Left, Right** → `$FFE1 = 1` (A-Reset)
3. Нажать **C** дважды, снова **Up, Down, Left, Right** → `$FFE2 = 1` (Debug)
4. Зажать **A**, нажать **Start** → Level Select
5. В Level Select зажать **A** при выборе уровня → `$FFFA = 1`
6. В игре зажать **C** → **Debug Placement Mode** (`$FE08 = 1`)

> **Вывод:** В Rev00 debug mode активируется **тройным** вводом D-pad чита с промежуточными нажатиями C, а **не** через Sound Test. Sound Test чит `01,09,09,01,00,06,02,03` существует только в **Rev01+**.

### 8.5 Z80 Sound Driver (SMPS)

#### Загрузка

```m68k
; SoundDriverLoad ($1352)
$1354  move.w     #$100, $a11100.l  ; Запрос Z80 Bus
$135C  move.w     #$100, $a11200.l  ; Z80 Reset ON
$1364  lea.l      $72e7c.l, a0      ; Источник: ROM $72E7C (Kosinski)
$136A  lea.l      $a00000.l, a1     ; Назначение: Z80 RAM $A00000
$1370  bsr.w      $189c             ; KosDec — декомпрессия
$1374  move.w     #$0, $a11200.l    ; Z80 Reset OFF (начинает выполнение)
$138C  move.w     #$0, $a11100.l    ; Освободить Z80 Bus
```

#### Регистры управления

| M68K адрес | Z80 адрес | Назначение |
|---|---|---|
| `$A11100` | — | Z80 Bus Request |
| `$A11200` | — | Z80 Reset |
| `$A00000`+ | `$0000`+ | Z80 RAM (8 КБ) |

#### Звуковые очереди (M68K → Z80)

Три очереди для отправки звуковых команд:

```m68k
; $1396 — PlaySound_Special (очередь 1)
$1396  move.b     d0, $f00a.w       ; Sound Queue 1

; $139C — PlaySound (основная, очередь 2)
$139C  move.b     d0, $f00b.w       ; Sound Queue 2

; $13A2 — PlaySound_2 (очередь 3)
$13A2  move.b     d0, $f00c.w       ; Sound Queue 3
```

> `$F00A`–`$F00C` — это RAM-ячейки, которые Z80 драйвер читает каждый кадр. M68K записывает туда ID звука, Z80 забирает и обрабатывает.

#### Инициализация Z80

Крохотный Z80 stub в начале ROM (`$02EA`):

```z80
$02EA: F3          DI              ; Отключить прерывания
$02EB: ED 56       IM 1            ; Interrupt Mode 1
```

### 8.13 Верификация «заглушек» объектов $02–$07

Все 6 заглушек указывают на `$D5B2` — это функция **ObjectFall**:

```m68k
$D5B2  move.l     $8(a0), d2        ; X position (long)
$D5B6  move.l     $c(a0), d3        ; Y position (long)
$D5BA  move.w     $10(a0), d0       ; X velocity
$D5BE  ext.l      d0
$D5C0  asl.l      #$8, d0
$D5C2  add.l      d0, d2            ; X += Xvel * 256
$D5C4  move.w     $12(a0), d0       ; Y velocity
$D5C8  addi.w     #$38, $12(a0)     ; Y velocity += $38 (гравитация!)
```

Это НЕ `DeleteObject`. Если создать объект с ID $02–$07:
- Ему **не назначается** спрайт или коллизия
- Гравитация (`+$38` за кадр к Y-скорости) заставляет объект **бесконечно падать**
- Объект занимает слот но никогда не удаляется самостоятельно → **утечка слотов**

> Это подтверждает что слоты $02–$07 были **зарезервированы** на этапе разработки но **никогда не реализованы**. Вероятно, для дополнительных режимов Sonic (Tails/Knuckles были запланированы позже) или альтернативных контроллеров.

### 8.14 Kosinski Decompressor ($1452)

Анализ декомпрессора Kosinski (используется для Z80 драйвера, демо-данных, и level art):

```m68k
$1452  movem.l    d0-d7/a0-a1/a3-a5, -(a7)  ; Сохранить все регистры
$1456  lea.l      $1518.l, a3       ; Internal decompression table
$145C  lea.l      $aa00.w, a1       ; RAM buffer ($AA00)
$1460  move.w     (a0)+, d2         ; Прочитать первое слово
$1462  lsl.w      #$1, d2           ; Бит 15: XOR-mode
$1464  bcc.b      $146a
$1466  adda.w     #$a, a3           ; Alternate table for XOR
$146A  lsl.w      #$2, d2           ; Сдвинуть для обработки
$146C  movea.w    d2, a5
$146E  moveq      #$8, d3           ; 8 тайлов за раз
$1470  moveq      #$0, d2
$1472  moveq      #$0, d4
$1474  bsr.w      $152e             ; Internal decode
$1478  move.b     (a0)+, d5         ; Прочитать data byte
$147A  asl.w      #$8, d5
$147C  move.b     (a0)+, d5         ; Second data byte
$147E  move.w     #$10, d6          ; 16 bits per tile row
```

> Kosinski — это LZ-based compression с Huffman-coded lengths. Используется во ВСЕХ играх Sonic для Genesis. SMPS Z80 драйвер ($72E7C → $A00000) — основной потребитель.
